Donkey MVP 用户手册

适用版本：本地可试用版本。

这份手册面向：想把一个需求、技术方案或验收任务交给 Donkey 处理的试点用户。它不讲 Donkey 如何开发，只讲当前工具怎么用、能交付什么、不能交付什么。

1. 当前能不能一句话开始开发需求？

可以进入本地代码修改链路，但不会自动提交 PR。Donkey 会先判断目标阶段；如果命中开发目标且已配置开发命令，会调用本地 Coding Agent Adapter 修改工作区，然后继续运行测试并生成证据包。Donkey 当前不会 commit、push、创建 PR、自动合入或上线。

你说一句话当前 Donkey 会做什么不会做什么
“我想优化内部任务筛选体验”识别为想法，默认生成需求文档草案和信息缺口。不会跳过需求边界直接改代码。
“请开发一个本地搜索功能并补充测试”识别为开发目标；已配置开发命令时，调用 Coding Agent Adapter 修改代码并跑测试。不会 commit、push 或创建 PR。
“已有技术方案，请按方案执行”识别为已有方案的开发目标；跳过需求梳理，调用 Coding Agent Adapter 写代码，再进入测试验收。不会创建分支或提交 PR。
“已有技术方案，请直接执行测试验收”跳过需求和开发执行，运行配置的测试命令，生成 HTML 证据包。不会声称已经完成开发。
“请修改 .env 文件并提交 PR”命中高危路径，降级到风险报告，阻断工具执行。不会执行测试、提交、推送或修改敏感文件。

效果评测已实现内置样本版运行 6 个内置 EvalCase，检查目标阶段判断和高危误放行。

3. 还没有完成的能力

能力当前状态意味着什么
无需配置即可稳定开发任意需求未完成当前依赖本地 Coding Agent CLI 和仓库测试质量，复杂需求仍需要人工验收。
团队级任务委派未完成当前是单机单次运行，不是多人队列或 Web Console。
自动创建 PR未完成当前不会提交代码、push 分支或创建 Pull Request。
CI Runner / Container Runner未完成当前只支持本地运行命令，不提供容器隔离和 CI 编排。
飞书入口 / Web Console未完成当前主要是 CLI、TUI 和本地 HTML 报告，不是面向全员的网页产品。
真实 B/D 需求闭环评测未完成内置样本还少，不能代表真实需求成功率。
稳定自动修复测试失败未完成当前会报告失败；是否继续修复由外部 Coding Agent 能力和人工确认决定。

4. 你应该怎么使用它

当前推荐把 Donkey 当作“本地自动开发与验收助手”，而不是完整团队研发平台。

使用场景推荐输入你应该看什么结果
把想法整理成需求“我想做一个 xxx，目标是 xxx”看需求草案是否覆盖目标、范围、非目标和信息缺口。
把需求推进到方案“这是需求文档，请输出技术方案：...”看技术方案草案是否有改动范围、风险和测试策略。
已有方案，只需要验收“已有技术方案，请直接执行测试验收”看 HTML 证据包里的测试命令、退出码、日志路径和建议结论。
已有方案，需要直接落地“已有技术方案，请按方案执行”看代码变更报告、测试报告、HTML 证据包和工作区 diff。
直接开发一个需求“请开发 xxx，并补充测试”看代码变更报告、测试报告和工作区 diff。
检查高危请求“请修改 .env / deploy / production 配置...”确认它是否降级到风险报告且没有执行工具命令。

5. 一次运行结束后怎么看报告

报告区域用户需要判断的问题
输入与目标Donkey 是否正确理解了你的输入类型和目标阶段。
Workflow 阶段哪些阶段被执行，哪些被跳过；是否存在不该跳过或不该执行的阶段。
工具执行实际运行了什么命令，是否通过，退出码是什么。
证据是否有测试报告、风险报告、HTML 证据包等可审阅产物。
建议结论接受、人工审阅还是阻断；最终仍由人确认。

6. 推荐输入模板

从想法开始

我想做【目标】，主要用户是【用户】，希望解决【问题】，先帮我整理需求文档。

从需求开始

这是需求文档：【粘贴内容】。请推进到技术方案，重点关注影响范围、风险和验收方式。

从技术方案开始

已有技术方案，请按方案执行。完成后运行 npm test 并生成验收报告。

只做测试验收

已有技术方案，请直接执行测试验收。测试命令使用 npm test。

只做风险判断

这个需求涉及【生产 / secret / .env / deploy / infra】，请先判断风险并输出风险报告。

7. 当前入口

当前没有飞书入口和 Web Console。试点阶段可通过 TUI 或短命令在仓库本地发起运行。

前置条件：本地已安装依赖并完成构建；如果要让 Donkey 写代码，需要在 TUI 中配置开发命令，或在 .donkey/repo-profile.json 中配置 commands.develop。

动作命令
打开 TUInpm start
一句话运行npm run d -- "请开发一个本地搜索功能并补充测试"
初始化 Donkey 配置npm run donkey -- init --repo . --json
发起一次运行npm run donkey -- run --repo . --input "..." --json
指定测试命令验收npm run donkey -- run --repo . --input "已有技术方案，请直接执行测试验收" --test-command "npm test" --json
查看状态npm run donkey -- status <runId> --repo . --json
查看报告路径npm run donkey -- show <runId> --repo .
运行内置评测npm run donkey -- eval --repo . --json

8. 使用边界

  • 测试通过不等于需求完成，只代表配置的验证命令通过。
  • 风险报告不是审批通过，它表示当前请求不应自动执行。
  • 代码修改发生在当前工作区；Donkey 不会自动 commit、push 或创建 PR。
  • 当前证据包在本地 .donkey/ 下，长期归档要沉淀到 docs/reviews/。
  • 不要把生产凭证、token、secret 明文输入给 Donkey。
  • “一句话自动开发到 PR”仍是后续目标；当前只做到本地代码修改、测试和证据包。
