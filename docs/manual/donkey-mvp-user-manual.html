Donkey MVP 使用手册

适用版本：0.1.0，本地 Phase 0/1 MVP。

适用人群：希望用 Donkey 在本地完成目标阶段判断、测试验收、风险降级和证据归档的技术基建团队成员。

1. 能做什么

能力当前支持当前边界
初始化仓库画像生成 Repo Profile 和 Agent Profiles。Repo Profile 仍需结合真实仓库继续补充。
判断目标阶段从自然语言输入判断应到需求文档、技术方案、任务拆解、测试验收、PR 或风险报告。当前是规则化实现，不代表完整语义理解。
测试验收按 Repo Profile 或命令行传入的测试命令运行本地验证。不做真实代码自动修改和 PR 创建。
风险降级高危关键词和高危路径命中后降级到风险报告，不执行工具命令。规则不是完整安全沙箱，不能替代人工审查。
证据归档生成 state、日志、Markdown 证据和 HTML 证据包。.donkey/ 为本地运行目录，不随 git 提交。
评测 replay运行内置 EvalCase，输出 JSON 与 HTML 报告。当前样本数量少，需要持续引入真实试点样本。

2. 环境准备

  • Node.js 20 或以上。
  • 已安装 npm。
  • 在仓库根目录执行命令。
npm install
npm run build
npm test

3. 初始化 Donkey

初始化会在 .donkey/ 下生成仓库画像和 Agent Profiles。

npm run donkey -- init --repo . --json
产物用途
.donkey/repo-profile.json记录测试命令、风险关键词、高危路径、命令 allowlist / blocklist。
.donkey/agent-profiles.json记录内置角色 Agent 的描述、工具、技能和权限边界。

4. 运行一次测试验收

当输入已经是技术方案，且目标是直接测试验收时，Donkey 会跳过需求梳理和开发执行，只运行验证并生成证据包。

npm run donkey -- run \
  --repo . \
  --input "已有技术方案，请直接执行测试验收" \
  --test-command "npm test" \
  --json

成功输出示例：

{"runId":"run-...","status":"completed","targetStage":"validation_report","reportPath":".donkey/runs/run-.../report.html"}

5. 查看运行状态和证据

npm run donkey -- status <runId> --repo . --json
npm run donkey -- show <runId> --repo .
字段说明
statuscompleted、blocked 或 failed。
targetStage本次自动推进到的目标阶段。
recommendedDecision建议接受、人工审阅或阻断。
reportPathHTML 证据包路径，供人审阅。

6. 风险 Gate 验证

高危路径或关键词命中后，Donkey 会生成风险报告，不执行测试命令或提交命令。

npm run donkey -- run \
  --repo . \
  --input "请修改 .env 文件并提交 PR" \
  --target-stage pull_request \
  --test-command "npm test" \
  --json

预期结果：status=blocked，targetStage=risk_report，toolRuns=[]。

7. 运行评测集

npm run donkey -- eval --repo . --json

当前内置评测集覆盖想法到需求、需求到技术方案、技术方案到测试验收、PR 到验收、高危降级。输出包含通过率、输入类型准确率、目标阶段准确率和高危误放行数。

8. 配置 Repo Profile

可按仓库实际情况编辑 .donkey/repo-profile.json。

配置项说明
commands.test默认测试命令。
risk.highRiskKeywords触发风险报告的关键词。
risk.highRiskPaths触发风险报告的路径，例如 .env、deploy、infra。
risk.allowedCommandPatterns允许执行的命令正则。未命中 allowlist 的命令默认拒绝。
risk.blockedCommandPatterns强制拒绝的命令正则。

Tool Gateway 会先拦截 shell 控制符和重定向，再匹配 allowlist，最后以 shell: false 执行 argv。

9. 常见问题

为什么已有技术方案不会执行开发？

当前 MVP 是 Local Runner 骨架，只承诺测试验收、风险降级和证据闭环。真实代码自动修改、外部 Coding Agent、PR 创建属于后续阶段。

为什么报告路径在 .donkey/ 下？

.donkey/ 是本地运行态目录，便于反复试跑。需要长期留档的结论应沉淀到 docs/reviews/。

测试通过是否等于需求完全正确？

不是。测试通过只说明配置的验证命令通过。证据包会保留边界说明，最终验收仍需要人判断交付结果是否符合预期。

10. 交付前检查

  • 运行 npm test。
  • 检查 README、CHANGELOG、使用手册和验收报告是否需要更新。
  • 确认没有提交 .donkey/、dist/、node_modules/ 或敏感信息。
  • 重要结论输出前，按仓库指令启动最高思考等级 reviewer 复查。
