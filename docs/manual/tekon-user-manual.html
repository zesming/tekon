SECTION 01

天工是什么

天工（Tekon）是一个本地 Agent workflow 驾驶系统——一个"受控研发工作台"。

天工（Tekon）不是聊天机器人，也不是自动上线平台。用户把一个低到中风险的研发需求交给天工，天工会按固定 workflow 拆成角色任务、在隔离 worktree 中执行、跑验证 gate、沉淀 artifact 和审计记录，最后整理出可审阅材料和 PR 准备包。

天工的核心目标是增强人类交付能力，而不是替人类做高风险决策。它可以帮助你把需求推进到"可以人工审阅、可以准备 PR、可以继续验证"的状态；合入、上线、权限扩大、生产变更仍然由人控制。

当前定位

本地 CLI/Web 工具

在你自己的机器上运行，不依赖远程服务。

受控 Workflow 执行器

按固定流程拆解、执行、验证研发任务。

证据与审阅收集器

统一收集 artifact、gate 日志、审计事件。

PR 准备助手

帮你生成可审阅材料和 PR 准备包。

研发工作样本评估器

评估真实样本是否达到试用门槛。

多 Provider 执行入口

支持 mock、Claude Code 和 Codex provider。

名称说明天工的英文名是 Tekon，取 Tech + Kong 的融合谐音，中文名取"天工"，源自《天工开物》。
SECTION 02

天工解决什么问题

五个核心痛点，天工逐一应对。

2.1 需求进入研发前不清楚

真实工作里，很多需求只有一句话。直接交给 Agent 容易出现边界不清、验收标准不清、风险不清。天工提供 demand shape，先把需求塑形成需求卡，包含：需求标题和正文、推荐 workflow 模板、风险等级和风险标签、非目标、开放问题、验收标准。用户可以先审阅和批准需求卡，再发起执行。

2.2 不知道该选什么 Workflow

standard-featurebugfixtest-improvementdocs-updateplan-onlystandard-delivery

standard-delivery 是标准交付治理流程，包含 PM 内审、PM/RD/QA 外部需求评审、RD 技术评审、QA 测试方案评审、独立变更评审、QA final signoff、QA signoff review 和 PMO checkpoint。workflow select 会给出推荐模板和理由；eval workflow-selection 会检查人工选择是否合理。

2.3 Agent 输出不可审阅

天工要求 provider 输出结构化 artifact，并把 gate、日志、审计事件和 PR 包统一组织。用户可通过 review 或 Web dashboard 查看：readiness 失败项、证据入口、Gate 失败诊断、影响文件和 diff、Artifact 正文预览、Gate 日志、PR 准备包、下一步命令建议。

2.4 远端副作用需要人控制

delivery dry-run → 只看交付计划 delivery prepare → 只生成本地 PR 包 delivery create-pr --approve-human → 人工批准后才 push delivery ci-status → 只读查询 PR checks delivery ci-watch → 只读轮询，不 rerun/merge

2.5 需要判断一次 run 是否真的可交付

eval readiness 评估单个 run 的交付证据是否完整。eval work-usability 评估一组真实样本是否达到试用门槛。

SECTION 03

核心用户场景

五种典型场景，从功能开发到工具评估。

SCENARIO A
我有一个小功能，希望推进到可审 PR

适用：给内部工具增加筛选条件、为 CLI 补低风险命令、给 Web dashboard 增加入口。

  1. tekon init 初始化目标仓库
  2. demand shape 把需求写成需求卡
  3. 人工审阅需求卡
  4. demand approve 批准需求卡
  5. run 发起 workflow
  6. status 和 review 查看结果
  7. delivery prepare 生成 PR 准备包
  8. 人工确认后 delivery create-pr --approve-human
  9. delivery ci-status 或 ci-watch 写回远端 CI 证据
  10. eval readiness 判断 PR/CI 证据是否完整
SCENARIO B
我只想修一个 bug，但需要人工确认风险

适用：修改已有逻辑分支、修复低风险接口问题、调整内部工具状态处理。

  1. 使用 bugfix 模板运行
  2. 如果触发 human gate，先执行 approval summary
  3. 用 eval approval-summary 检查审批摘要是否完整
  4. 人工判断：resume --approve-human 批准继续，或 approval reject 拒绝并阻断
SCENARIO C
我只想补测试

适用：为模块补单测、为失败路径补回归测试、为 CLI/Web API 增加覆盖。

  1. workflow select 确认是否推荐 test-improvement
  2. run --template test-improvement 执行
  3. 看 gate 是否通过
  4. 用 review 检查 artifact 和测试证据
SCENARIO D
我只想写文档或方案

适用：更新用户手册、整理验收报告、写技术方案或产品方案。

docs-update — 文档更新plan-only — 只做方案standard-delivery — 验证完整角色链路

standard-delivery 当前适合 Tekon 自身 dogfooding 和低风险种子任务。

SCENARIO E
我要判断天工是否已经能用于真实工作
  1. 挑选 2~3 个真实但低风险的仓库
  2. 准备 10 个真实或历史需求
  3. 每次 run 后用 eval work-usability record 写入样本清单
  4. 用 eval work-usability --samples 评估样本集
  5. 把 Markdown/HTML 报告保存到 docs/reviews/
SECTION 04

Quick Start

以下示例假设 tekon 已安装到 PATH，并且你正在目标项目根目录执行命令。

命令约定没有全局安装时，可用 node ~/.tekon/packages/cli/dist/index.js 替换 tekon；从其它目录操作目标仓库时，再显式追加 --repo /path/to/project。

4.1 安装

⬇ 一键安装（推荐）
curl -fsSL https://raw.githubusercontent.com/zesming/tekon/main/scripts/install.sh | bash

安装脚本会自动完成：克隆仓库 → 安装依赖 → 构建项目 → 配置 tekon 命令到 PATH。

安装完成后，执行 source ~/.zshrc（或 ~/.bashrc）使 PATH 生效，即可在任意目录使用：

tekon

如需指定安装目录或分支：

TEKON_HOME=/opt/tekon TEKON_VERSION=main curl -fsSL https://raw.githubusercontent.com/zesming/tekon/main/scripts/install.sh | bash

如果不想配置 PATH，也可以直接用 node 执行：

node ~/.tekon/packages/cli/dist/index.js
前置依赖需要 git、node（>=18）、npm。安装脚本会自动检测，缺少时会给出安装指引。

本文默认使用 tekon 示例，重点展示普通用户的最短路径。

4.2 初始化目标仓库

tekon init

初始化会在目标仓库生成 .tekon/ 运行态目录，包含配置、数据库、工作区、workflow、角色和 Web session token。

4.3 检查目标仓库命令画像

tekon workflow preflight
状态含义
resolved该 gate 命令已解析
missing目标仓库缺少对应命令，需要补 repo profile
not-applicable用户显式声明不适用
not-command-gateschema、role-scope、QA signoff 等语义 gate，不需要 repo profile 命令
suggestedCommand天工从 package.json 推断的候选命令

4.4 塑形需求

tekon demand shape "给 Web dashboard 增加审批摘要展示，要求 e2e 通过"

命令会输出 shapePath 和 reviewPath。先读 Markdown 审阅稿，确认需求边界后批准：

tekon demand approve

可选：评估需求卡质量。

tekon eval demand-shape

4.5 发起运行

tekon run

输出里会有 runId。后续常规命令默认读取最近一次 run。

长程任务可显式放大外层预算：

tekon run --timeout-ms 7200000 --no-progress-timeout-ms 1200000 --progress-heartbeat-ms 30000

未传 --template 时默认运行 standard-delivery；未传 --agent 时默认使用 Codex provider。离线回归可显式切到 mock：

tekon run --template standard-delivery --agent mock
Codex provider 说明Codex 使用本机 codex --profile internal --sandbox workspace-write --ask-for-approval on-request --add-dir <TEKON_OUTPUT_DIR> exec，通过 TEKON_OUTPUT_DIR / $TEKON_ARTIFACT_MANIFEST 写回 artifact。真实 provider 默认总超时 1 小时，无进展超时 15 分钟。QA validation 会记录 tested ref，QA signoff/PR package/readiness 会校验所测即所得。

4.6 查看结果

tekon status
tekon review
提示PR/CI 证据尚未写回前，eval readiness 通常会因 pr-prepared、pr-created 或 remote-ci-passed 失败而保持 ready=false。

4.7 准备 PR 材料

tekon delivery prepare

当前只支持 standard-delivery 治理 run，只生成本地 PR 包，不 push。生成前会执行 pre-PR readiness 校验。

4.8 创建远端 PR

tekon delivery create-pr --approve-human
注意产生真实远端副作用：push 分支并调用 GitHub CLI 创建 PR。不会绕过 pre-PR readiness、QA signoff 和所测即所得校验。

4.9 查询远端 CI

tekon delivery ci-status
tekon delivery ci-watch --max-attempts 20 --interval-ms 15000
tekon eval readiness

4.10 默认上下文规则

天工的常规 CLI 使用方式是"进入目标仓库根目录后执行短命令"。

Repo

优先使用 --repo；不传时从当前目录向上查找 .tekon/config.yaml，找不到时使用当前 Git 仓库根目录。

Demand shape

demand shape 默认写入 .tekon/demands/；demand approve 默认批准最近需求卡。

Run

run 没有需求文本时默认读取最近已批准需求卡；status、review、eval readiness、delivery prepare 默认使用最近一次 run。

Human decision

approval summary、approval reject、resume --approve-human 默认使用最近的 pending human decision。

显式传参的三种情况1. 从其它目录操作目标仓库时传 --repo；2. 查看历史对象时传 --run-id 等；3. 高危动作保留 --approve-human。
SECTION 05

核心概念

理解天工的十一个基础构件。

5.1 Repo

目标仓库，即你希望天工处理的项目。天工自身仓库和目标仓库可以相同，也可以不同。

5.2 .tekon/

天工在目标仓库中的运行态目录。保存配置、SQLite 数据库、run artifact、gate 日志、worktree 等。通常不提交。

5.3 Run

一次 workflow 执行。每个 run 有唯一 runId。查看状态、审阅、准备 PR 都围绕 run id。

5.4 Workflow

一组有顺序和依赖的角色节点。内置模板：standard-feature、bugfix、test-improvement、docs-update、plan-only、standard-delivery。

5.5 Role

执行节点的角色：PM、RD、QA、Reviewer、PMO。决定 prompt、知识和工具策略。

5.6 Gate

验证关卡：build、lint、test、schema、security-scan、human、independent-review、role-scope、ac-evidence、qa-signoff、process-completeness。

5.7 Provider

执行节点的 agent 后端：mock、claude-code、codex。真实 provider 需提供 artifact manifest。

5.8 Artifact

结构化产物：需求卡、代码变更说明、测试报告、审阅报告、PR 包、CI 状态。人工审阅和自动评估的主要证据。

5.9 Review Surface

聚合审阅面。CLI 命令是 review，Web dashboard 也使用同一套数据。

5.10 Readiness

单次 run 的工作就绪度评估。回答："这次 run 的 workflow、gate、artifact、PR、CI 证据是否完整？"

5.11 Work Usability

样本集级评估。回答："天工是否已在足够多真实样本上表现稳定？"

Workflow 执行流程概览

需求 → demand shape → 需求卡 → approve ↓ workflow select ↓ ┌──────── run ────────┐ │ PM → RD → QA → Reviewer → PMO │ │ ↓ ↓ ↓ ↓ ↓ │ │ gate gate gate gate gate │ └──────────────────────────┘ ↓ review → delivery prepare → create-pr → ci-watch → eval readiness

SECTION 07

Web Dashboard

本地 Web dashboard，用于审阅和人工决策。

启动

TEKON_PROJECT_ROOT=/path/to/project npm exec --yes -- pnpm@10.12.1 --filter @tekon/web dev

Web dashboard 适合：查看项目 overview、选择历史 run、查看 readiness/evidence/diff/artifact/gate log/PR 包、处理 human approval、发起受控模板 run、触发 delivery prepare 和 delivery create-pr。

写操作需要 session token写操作依赖 init 生成的 .tekon/web-session.json。没有 token 时写操作会被拒绝。token 不应提交。
SECTION 08

如何判断结果是否可信

不要只看"命令退出 0"。建议按顺序看。

  1. status：workflow 是否 passed。
  2. review：失败项和证据是否能解释。
  3. Changed Files：影响文件是否符合预期。
  4. Artifacts：需求、变更、测试、审阅证据是否完整。
  5. Gate Logs：build/lint/test/security 是否真的跑过。
  6. PR Package：PR body 是否能让 reviewer 看懂。
  7. CI Status：远端 checks 是否已记录。
  8. eval readiness：PR/CI 证据是否完整。
重要如果其中任何一步说不清楚，不要继续创建 PR 或批准高风险动作。
SECTION 09

常见问题处理

9.1 workflow preflight 显示 missing command

原因：目标仓库没有配置对应命令。处理：看 suggestedCommand → 写入 .tekon/repo-profile.yaml → 再跑 workflow preflight。

9.2 run 拒绝 dirty base

原因：有未提交业务改动。处理：先提交/stash，或追加 --allow-dirty-base。

9.3 pending human gate

处理：approval summary → eval approval-summary → resume --approve-human 或 approval reject。

9.4 readiness 不通过

常见原因：workflow 没 passed、pending gate、验证 gate 失败、security scan 失败。处理：先看 review 的 failed checks。

9.5 delivery create-pr 失败

常见原因：没有 gh、认证失败、工作区 dirty、PR 已存在。

9.6 ci-status 查询失败

常见原因：run 没有 PR URL、selector 不对。处理：用 --selector 明确指定。

9.7 Artifact 被拒绝入库

原因：产物命中密钥模式。处理：删除密钥内容，使用安全摘要。

9.8 Web 写操作被拒绝

处理：重新执行 init 生成 token → 确认 TEKON_PROJECT_ROOT 正确。

9.9 Codex provider 运行失败

处理：先验证 codex --version → 查看 .tekon/runs/<runId>/<nodeId>/ 下产物。不要把失败降级成 mock 通过。

SECTION 11

使用说明

天工的核心设计理念是增强人类交付，合入、上线、权限扩大等关键决策始终由人控制。

适用场景

内部工具研发效能工具测试补齐文档补齐低风险 bugfix可回滚的中小需求

手册更新规则

后续每次功能、CLI/Web 入口、参数、错误处理、边界或用户流程变化后，都必须评估是否需要更新本手册。当前主手册路径：docs/manual/tekon-user-manual.md
