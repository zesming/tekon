SECTION 01

天工是什么

天工（Tekon）是一个本地 Agent workflow 框架——一个"受控研发工作台"。

天工（Tekon）是一个本地 Agent workflow 框架——一个"受控研发工作台"：用户把研发需求交给天工，天工会按固定 workflow 拆成角色任务、在隔离 worktree 中执行、跑验证 gate、沉淀 artifact 和审计记录，最后整理出可审阅材料和 PR 准备包。

天工的核心目标是增强人类交付能力。它帮助你把需求推进到"可以人工审阅、可以准备 PR、可以继续验证"的状态；合入、上线、权限扩大、生产变更仍然由人控制。

当前定位

本地 CLI/Web 工具

在你自己的机器上运行，不依赖远程服务。

受控 Workflow 执行器

按固定流程拆解、执行、验证研发任务。

证据与审阅收集器

统一收集 artifact、gate 日志、审计事件。

PR 准备助手

帮你生成可审阅材料和 PR 准备包。

研发工作样本评估器

评估真实样本是否达到试用门槛。

多 Provider 执行入口

支持 mock、Claude Code 和 Codex provider。

名称说明天工的英文名是 Tekon，取 Tech + Kong 的融合谐音，中文名取"天工"，源自《天工开物》。
SECTION 02

天工解决什么问题

五个核心痛点，天工逐一应对。

2.1 需求进入研发前不清楚

真实工作里，很多需求只有一句话。直接交给 Agent 容易出现边界不清、验收标准不清、风险不清。天工提供 demand shape，先把需求塑形成需求卡，包含：需求标题和正文、推荐 workflow 模板、风险等级和风险标签、非目标、开放问题、验收标准。用户可以先审阅和批准需求卡，再发起执行。

2.2 不知道该选什么 Workflow

standard-featurebugfixtest-improvementdocs-updateplan-onlystandard-delivery

standard-delivery 是标准交付治理流程，包含 PM/RD/QA/Reviewer/PMO 完整角色链路、独立评审、AC evidence、QA signoff 和流程完整性 gate。workflow select 会给出推荐模板和理由；eval workflow-selection 会检查人工选择是否合理。

2.3 Agent 输出不可审阅

很多 Agent 工具会把结果散落在对话、文件和日志里。天工要求 provider 输出结构化 artifact，并把 gate、日志、审计事件和 PR 包统一组织起来。用户可以通过 review 或 Web dashboard 看：readiness 失败项、证据入口、Gate 失败诊断、影响文件和 diff、Artifact 正文预览、Gate 日志、PR 准备包、下一步命令建议。

2.4 远端副作用需要人控制

真实 push、创建 PR、等待远端 CI 都属于有副作用或外部依赖的动作。天工把这些动作拆开：

2.5 需要判断一次 run 是否真的可交付

eval readiness 会评估单个 run 的交付证据是否完整。当前 pr-prepared、pr-created 和 remote-ci-passed 都是 required，因此在 PR 准备、真实 PR 创建或远端 CI 证据写回之前，ready=false 是预期状态。eval work-usability 会评估一组真实样本是否达到试用门槛，避免只靠 fixture 或 demo 宣称可用。

SECTION 03

核心用户场景

四种典型场景。实线为必须操作，虚线框为人类可选观察（不影响流程推进）。

SCENARIO A
我有一个小功能，希望推进到可审 PR

适用：给内部工具增加筛选条件、为 CLI 补低风险命令、给 Web dashboard 增加入口。

完整交付流程 — 从需求到可审 PR（Human ↔ Tekon 交替时序）
人工必须操作 Tekon 自动执行 可选观察
SCENARIO B
我只想修一个 bug，但需要人工确认风险

适用：修改已有逻辑分支、修复低风险接口问题、调整内部工具状态处理。

Bug 修复 + 人工审批流程
人工必须操作 Tekon 自动执行
SCENARIO C
我只想补测试

适用：为模块补单测、为失败路径补回归测试、为 CLI/Web API 增加覆盖。

测试补齐流程 — 人类观察步骤均为可选
人工必须操作 Tekon 自动执行 可选观察（虚框）
SCENARIO D
我只想写文档或方案

适用：更新用户手册、整理验收报告、写技术方案或产品方案。

docs-update — 文档更新plan-only — 只做方案，不执行代码standard-delivery — 完整角色链路验证

standard-delivery 适合 Tekon 自身 dogfooding 和低风险种子任务。

SECTION 04

Quick Start

以下示例假设 tekon 已安装到 PATH，并且你正在目标项目根目录执行命令。

⬇ 一键安装
curl -fsSL https://raw.githubusercontent.com/zesming/tekon/main/scripts/install.sh | bash

安装脚本自动完成：克隆仓库 → 安装依赖 → 构建项目 → 配置 tekon 到 PATH。执行 source ~/.zshrc（或 source ~/.bashrc）即可使用。前置依赖：git、node (>=18)、npm。

4.2 初始化目标仓库

tekon init

初始化会在目标仓库生成 .tekon/ 运行态目录，包含配置、数据库、工作区、workflow、角色和 Web session token。

4.3 检查命令画像

tekon workflow preflight
状态含义
resolved该 gate 命令已解析
missing目标仓库缺少对应命令
not-applicable用户显式声明不适用
not-command-gateschema、role-scope、QA signoff 等语义 gate
suggestedCommand天工从 package.json 推断的候选命令

4.4 塑形需求

tekon demand shape "给 Web dashboard 增加审批摘要展示，要求 e2e 通过"
tekon demand approve
tekon eval demand-shape

4.5 发起运行

tekon run
# 长程任务超时控制
tekon run --timeout-ms 7200000 --no-progress-timeout-ms 1200000
# 使用 mock provider 离线回归
tekon run --template standard-delivery --agent mock
Provider 说明默认使用 Codex provider。真实 provider 通过 TEKON_OUTPUT_DIR 写回结构化 artifact，默认总超时 1 小时，可用 --timeout-ms 等参数覆盖。远端副作用仍由 delivery create-pr --approve-human 人工批准后执行。

4.6 查看结果

tekon status
tekon review

4.7 准备 PR

tekon delivery prepare

当前只支持 standard-delivery 治理 run，只生成本地 PR 包，生成前会执行 pre-PR readiness 校验。

4.8 创建远端 PR

tekon delivery create-pr --approve-human

4.9 查询远端 CI

tekon delivery ci-status
tekon delivery ci-watch --max-attempts 20 --interval-ms 15000
tekon eval readiness

4.10 默认上下文规则

天工的常规 CLI 使用方式是"进入目标仓库根目录后执行短命令"。默认推断规则：--repo 从当前目录向上查找 .tekon/config.yaml；demand shape/approve 默认操作最近需求卡；run/status/review/delivery prepare 默认使用最近一次 run；approval summary/approve/reject/resume 默认使用最近的 pending human decision。

显式传参的三种情况1. 跨目录操作时传 --repo；2. 查看历史对象时传 --run-id 等；3. 高危动作保留 --approve-human。
SECTION 05

核心概念

理解天工的十一个基础构件。

5.1 Repo

目标仓库，即你希望天工处理的项目。天工自身仓库和目标仓库可以相同，也可以不同。

5.2 .tekon/

运行态目录。保存配置、SQLite 数据库、run artifact、gate 日志、worktree 等。通常不提交。

5.3 Run

一次 workflow 执行。每个 run 有唯一 runId。查看状态、审阅、准备 PR 都围绕 run id。

5.4 Workflow

一组有顺序和依赖的角色节点。内置模板：standard-feature、bugfix、test-improvement、docs-update、plan-only、standard-delivery。

5.5 Role

执行节点的角色：PM、RD、QA、Reviewer、PMO。角色决定 prompt、知识和工具策略。

5.6 Gate

验证关卡：build、lint、test、schema、security-scan、human、independent-review、role-scope、ac-evidence、qa-signoff、process-completeness。

5.7 Provider

执行节点的 agent 后端：mock、claude-code、codex。真实 provider 需提供 artifact manifest。

5.8 Artifact

结构化产物：需求卡、代码变更说明、测试报告、审阅报告、PR 包、CI 状态。

5.9 Review Surface

聚合审阅面。CLI 命令是 review，Web dashboard 也使用同一套数据。

5.10 Readiness

单次 run 的工作就绪度评估。回答："workflow、gate、artifact、PR、CI 证据是否完整？"

5.11 Work Usability

样本集级评估。回答："天工是否已在足够多真实样本上表现稳定？"

SECTION 06

命令详解

天工 CLI 全部命令的参数、用法和故障处理。

6.1 init初始化目标仓库
tekon init
参数
  • --repo <path> — 从其它目录初始化指定仓库时使用
6.2 workflow preflight运行前检查命令画像
tekon workflow preflight
判断结果
  • resolved — 可执行
  • missing — 缺命令
  • not-applicable — 已声明不适用
  • not-command-gate — 语义 gate
6.3 workflow select根据需求推荐模板
tekon workflow select "补齐 CLI 单元测试"
6.4 demand shape需求塑形成需求卡
tekon demand shape "需求文本"
参数
  • --no-write — 只预览
  • --format json — 输出 JSON
6.5 demand approve人工批准需求卡
tekon demand approve
6.6 run发起一次 workflow
# 模板运行
tekon run "需求文本" --template standard-delivery --agent mock

# 需求卡运行（默认读取最近已批准需求卡）
tekon run

# 动态 dry-run
tekon run --dynamic --dry-run "需求文本"
参数
  • --template <name> — 使用内置模板
  • --agent mock|claude-code|codex — 选择 provider
  • --allow-dirty-base — 允许基于未提交改动运行
  • --timeout-ms / --no-progress-timeout-ms / --progress-heartbeat-ms — 超时控制
6.7 status查看 run 当前状态
tekon status
6.8 approval summary生成可复制审批摘要
tekon approval summary
参数
  • --run-id / --decision-id — 指定审批项
  • --json — 输出结构化 JSON
6.9 eval approval-summary检查审批摘要完整性
tekon eval approval-summary
6.10 resume --approve-human批准 human gate 并继续
tekon resume --approve-human
6.11 approval reject拒绝并阻断 workflow
tekon approval reject
6.12 review看完整审阅材料
tekon review
重点章节
  • Readiness Failed Checks / Evidence Navigation / Gate Failure Triage
  • Delivery / Changed Files / Artifacts / Gate Logs / PR Body / PR Package
6.13 delivery dry-run只看交付计划，无副作用
tekon delivery dry-run
6.14 delivery prepare生成本地 PR 准备包
tekon delivery prepare
结果
  • pr-package.md / pr-body.md
  • delivery-package artifact + 审计事件
6.15 delivery create-pr人工批准后创建远端 PR
tekon delivery create-pr --approve-human
必要条件
  • 已安装并认证 gh CLI
  • workflow passed、AC evidence 完整、QA signoff 满足
6.16 delivery ci-status只读查询 PR checks
tekon delivery ci-status
6.17 delivery ci-watch轮询 PR checks 直到终态
tekon delivery ci-watch --max-attempts 20 --interval-ms 15000
6.18 eval readiness判断单次 run 交付证据是否完整
tekon eval readiness
6.19 eval work-usability评估样本集是否达到试用门槛
tekon eval work-usability --samples /path/to/samples.yaml
SECTION 07

Web Dashboard

本地 Web dashboard，用于审阅和人工决策。

启动

TEKON_PROJECT_ROOT=/path/to/project npm exec --yes -- pnpm@10.12.1 --filter @tekon/web dev

Web dashboard 适合：查看项目 overview、选择历史 run、查看 readiness/evidence/diff/artifact/gate log/PR 包、处理 human approval、发起受控模板 run、触发 delivery prepare 和 delivery create-pr。

Session Token写操作依赖 init 生成的 .tekon/web-session.json。没有 token 时写操作会被拒绝。token 不应提交。

Web 是本地 dashboard，不是远程服务。Web create-pr 和 CLI 一样，未批准时只落库等待审批，批准后才 push 和创建 PR。

SECTION 08

如何判断结果是否可信

不要只看"命令退出 0"。建议按顺序看。

  1. 1. status：workflow 是否 passed。
  2. 2. review：失败项和证据是否能解释。
  3. 3. Changed Files：影响文件是否符合预期。
  4. 4. Artifacts：需求、变更、测试、审阅证据是否完整。
  5. 5. Gate Logs：build/lint/test/security 是否真的跑过。
  6. 6. PR Package：PR body 是否能让 reviewer 看懂。
  7. 7. CI Status：远端 checks 是否已记录。
  8. 8. eval readiness：PR/CI 证据是否完整。
重要如果其中任何一步说不清楚，不要继续创建 PR 或批准高风险动作。
SECTION 09

常见问题处理

9.1 workflow preflight 显示 missing command

原因：目标仓库没有配置对应命令。处理：看 suggestedCommand → 写入 .tekon/repo-profile.yaml → 再跑 workflow preflight。

9.2 run 拒绝 dirty base

原因：有未提交业务改动。处理：先提交/stash，或追加 --allow-dirty-base。

9.3 pending human gate

处理：approval summary → eval approval-summary → resume --approve-human 或 approval reject。

9.4 readiness 不通过

常见原因：workflow 没 passed、pending gate、验证 gate 失败、security scan 失败。处理：先看 review 的 failed checks。

9.5 delivery create-pr 失败

常见原因：没有 gh、认证失败、工作区 dirty、PR 已存在。

9.6 ci-status 查询失败

常见原因：run 没有 PR URL、selector 不对。处理：用 --selector 明确指定。

9.7 Artifact 被拒绝入库

原因：产物命中密钥模式。处理：删除密钥内容，使用安全摘要。

9.8 Web 写操作被拒绝

处理：重新执行 init 生成 token → 确认 TEKON_PROJECT_ROOT 正确。

9.9 Codex provider 运行失败

处理：先验证 codex --version → 查看 .tekon/runs/<runId>/<nodeId>/ 下产物。不要把失败降级成 mock 通过。

SECTION 10

参数速查

全局常见参数

参数用途
--repo <path>跨仓库操作时指定目标仓库
--run-id <runId>指定历史 run
--agent mock|claude-code|codex选择 provider
--approve-human明确批准人工 gate 或远端副作用
--allow-dirty-base允许基于未提交改动运行
--shape <path>指定需求卡路径
--decision-id <id>指定人工决策

run 参数

参数用途
--template <name>使用内置模板
--demand-file <path>指定需求卡
--dynamic / --dry-run动态 workflow 预览
--timeout-ms <ms>覆盖 provider 总超时
--no-progress-timeout-ms <ms>覆盖无进展超时
--progress-heartbeat-ms <ms>覆盖 heartbeat 间隔

delivery ci-watch 参数

参数用途
--selector <prUrl或branch>指定 PR 或分支
--max-attempts <n>最大轮询次数
--interval-ms <ms>每次轮询间隔
--backoff <n>轮询退避倍率
