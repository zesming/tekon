SECTION 01

天工是什么

天工（Tekon）是一个本地 Agent workflow 驾驶系统——一个"受控研发工作台"。

天工（Tekon）不是聊天机器人，也不是自动上线平台。用户把一个低到中风险的研发需求交给天工，天工会按固定 workflow 拆成角色任务、在隔离 worktree 中执行、跑验证 gate、沉淀 artifact 和审计记录，最后整理出可审阅材料和 PR 准备包。

天工的核心目标是增强人类交付能力，而不是替人类做高风险决策。它可以帮助你把需求推进到"可以人工审阅、可以准备 PR、可以继续验证"的状态；合入、上线、权限扩大、生产变更仍然由人控制。

当前定位

本地 CLI/Web 工具

在你自己的机器上运行，不依赖远程服务。

受控 Workflow 执行器

按固定流程拆解、执行、验证研发任务。

证据与审阅收集器

统一收集 artifact、gate 日志、审计事件。

PR 准备助手

帮你生成可审阅材料和 PR 准备包。

研发工作样本评估器

评估真实样本是否达到试用门槛。

多 Provider 执行入口

支持 mock、Claude Code 和 Codex provider；真实 provider 必须通过 artifact manifest 写回证据。

当前不是

生产级远程多租户平台 自动 merge / 自动上线 完整 DLP / 安全审计 OS 级沙箱 飞书 IM 通知机器人
名称说明 天工的英文名是 Tekon，取 Tech + Kong 的融合谐音，中文名取"天工"，源自《天工开物》。

SECTION 02

天工解决什么问题

五个核心痛点，天工逐一应对。

2.1 需求进入研发前不清楚

真实工作里，很多需求只有一句话："帮我补个功能""修一下这个问题"。直接交给 Agent 容易出现边界不清、验收标准不清、风险不清。

天工提供 demand shape，先把需求塑形成需求卡：

  • 需求标题和正文
  • 推荐 workflow 模板
  • 风险等级和风险标签
  • 非目标
  • 开放问题
  • 验收标准

用户可以先审阅和批准需求卡，再发起执行。

2.2 不知道该选什么 Workflow

不同工作不应该都套同一个流程。天工提供受控模板推荐：

standard-feature bugfix test-improvement docs-update plan-only standard-delivery

standard-delivery 是标准交付治理流程，包含 PM 内审、PM/RD/QA 外部需求评审、RD 技术评审、QA 测试方案评审、QA final signoff、QA signoff review 和 PMO checkpoint；当前已具备独立评审、角色范围、AC evidence、QA signoff、节点级 PMO checkpoint 和流程完整性 gate 的本地强约束，但不替代真实人类业务决策。

workflow select 会给出推荐模板和理由；eval workflow-selection 会检查人工选择是否合理。

2.3 Agent 输出不可审阅

很多 Agent 工具会把结果散落在对话、文件和日志里。天工要求 provider 输出结构化 artifact，并把 gate、日志、审计事件和 PR 包统一组织起来。用户可以通过 review 或 Web dashboard 看：

  • readiness 失败项
  • 证据入口
  • Gate 失败诊断
  • 影响文件和 diff
  • Artifact 正文预览
  • Gate 日志
  • PR 准备包
  • 下一步命令建议

2.4 远端副作用需要人控制

真实 push、创建 PR、等待远端 CI 都属于有副作用或外部依赖的动作。天工把这些动作拆开：

delivery dry-run → 只看交付计划，无副作用 delivery prepare → 只生成本地 PR 包 delivery create-pr → 人工明确批准后才 push --approve-human delivery ci-status → 只读查询 PR checks delivery ci-watch → 只读轮询，不 rerun/merge

2.5 需要判断一次 run 是否真的可交付

eval readiness 会评估单个 run 的交付证据是否完整。当前 pr-prepared、pr-created 和 remote-ci-passed 都是 required，因此在 PR 准备、真实 PR 创建或远端 CI 证据写回之前，ready=false 是预期状态。eval work-usability 会评估一组真实样本是否达到试用门槛，避免只靠 fixture 或 demo 宣称可用。

SECTION 03

核心用户场景

五种典型场景，从功能开发到工具评估。

SCENARIO A
我有一个小功能，希望推进到可审 PR

适用：给内部工具增加筛选条件、为 CLI 补低风险命令、给 Web dashboard 增加入口。

  1. tekon init 初始化目标仓库
  2. demand shape 把需求写成需求卡
  3. 人工审阅需求卡
  4. demand approve 批准需求卡
  5. run 发起 workflow
  6. status 和 review 查看结果
  7. delivery prepare 生成 PR 准备包
  8. 人工确认后 delivery create-pr --approve-human
  9. delivery ci-status 或 ci-watch 写回远端 CI 证据
  10. eval readiness 判断 PR/CI 证据是否完整
SCENARIO B
我只想修一个 bug，但需要人工确认风险

适用：修改已有逻辑分支、修复低风险接口问题、调整内部工具状态处理。

  1. 使用 bugfix 模板运行
  2. 如果触发 human gate，先执行 approval summary
  3. 用 eval approval-summary 检查审批摘要是否完整
  4. 人工判断：resume --approve-human 批准继续，或 approval reject 拒绝并阻断
SCENARIO C
我只想补测试

适用：为模块补单测、为失败路径补回归测试、为 CLI/Web API 增加覆盖。

  1. workflow select 确认是否推荐 test-improvement
  2. run --template test-improvement 执行
  3. 看 gate 是否通过
  4. 用 review 检查 artifact 和测试证据
SCENARIO D
我只想写文档或方案

适用：更新用户手册、整理验收报告、写技术方案或产品方案。

docs-update — 文档更新 plan-only — 只做方案，不执行代码改动 standard-delivery — 验证完整角色链路

standard-delivery 当前适合 Tekon 自身 dogfooding 和低风险种子任务，不适合直接承诺生产级强治理。

SCENARIO E
我要判断天工是否已经能用于真实工作
  1. 挑选 2~3 个真实但低风险的仓库
  2. 准备 10 个真实或历史需求
  3. 每次 run 后用 eval work-usability record 写入样本清单
  4. 用 eval work-usability --samples 评估样本集
  5. 把 Markdown/HTML 报告保存到 docs/reviews/

SECTION 04

Quick Start

以下示例假设 tekon 已安装到 PATH，并且你正在目标项目根目录执行命令。

命令约定 没有全局安装时，可用 node /path/to/tekon/packages/cli/dist/index.js 替换 tekon；从其它目录操作目标仓库时，再显式追加 --repo /path/to/project。

4.1 安装和构建

npm exec --yes -- pnpm@10.12.1 install --frozen-lockfile
npm exec --yes -- pnpm@10.12.1 build

构建后 CLI 路径：

node packages/cli/dist/index.js

如果将 CLI 安装到 PATH，命令名是 tekon。本文默认使用 tekon 示例，重点展示普通用户的最短路径。

4.2 初始化目标仓库

tekon init

初始化会在目标仓库生成 .tekon/ 运行态目录，包含配置、数据库、工作区、workflow、角色和 Web session token。

4.3 检查目标仓库命令画像

tekon workflow preflight

重点看：

状态 含义
resolved 该 gate 命令已解析
missing 目标仓库缺少对应命令，需要补 repo profile
not-applicable 用户显式声明不适用
not-command-gate schema、role-scope、QA signoff 等语义 gate 不需要 repo profile 命令
suggestedCommand 天工从 package.json 推断的候选命令

4.4 塑形需求

tekon demand shape "给 Web dashboard 增加审批摘要展示，要求 e2e 通过"

命令会输出 shapePath 和 reviewPath。先读 Markdown 审阅稿，确认需求边界后批准： 

tekon demand approve

可选：评估需求卡质量。

tekon eval demand-shape

4.5 发起运行

tekon run

输出里会有 runId。后续常规命令默认读取最近一次 run；只有查看历史 run 或避免歧义时才需要手动传 --run-id。

明确长程任务可以在 run 级别显式放大外层预算，例如 2 小时总超时、20 分钟无输出进展超时、30 秒 heartbeat： 

tekon run --timeout-ms 7200000 --no-progress-timeout-ms 1200000 --progress-heartbeat-ms 30000

未传 --template 时默认运行 standard-delivery；未传 --agent 时默认使用 Codex provider。离线回归或演示时，可显式切到 mock provider： 

tekon run --template standard-delivery --agent mock
Codex provider Codex provider 使用本机 codex exec 非交互模式，固定 codex --profile internal --sandbox workspace-write --ask-for-approval on-request --add-dir <TEKON_OUTPUT_DIR> exec，并通过 TEKON_OUTPUT_DIR / $TEKON_ARTIFACT_MANIFEST 写回 Tekon artifact；TEKON_ARTIFACT_MANIFEST 是 manifest 文件路径，不是字面文件名。 --add-dir 只由 Tekon 受控追加到本节点 artifact 输出目录。非 code-changes 节点会被要求只写节点 artifact、不修改仓库工作区，Engine 会在 worktree finalize 前拦截这类节点的源码变更；所有需要 artifact 的节点先写 artifact/manifest，再立即退出，不在节点内启动嵌套 subagent 审阅。 节点内也不执行 git add、git commit、git push 或创建 PR。结构化 JSON artifact 必须包含非空 title 和 body 字段；demand-card/prd 应使用 acceptanceCriteria[].id/description，有效的 acceptance_criteria[].criterion 会被兼容归一化；code-changes 的 provider-style JSON 如果包含非空 summary，或包含有效 changedFiles/verification 条目，可被归一化为可审阅 artifact，但真实 provider 仍应优先按 Tekon schema 写完整字段。test-plan 应使用 testBasis 和 testCases，有效的 sourceArtifactsReviewed/testScenarios 会被窄归一化。test-report/ac-evidence 应使用 criteriaEvidence[].criterionId/status/evidence，status 只能是 passed、failed、blocked 或 unknown，test-report.summary 如存在必须是字符串；若真实 provider 写出对象形式 summary，或用 id、evidenceSummary、coverage、passed_with_*/failed_with_*/blocked_with_* 状态标签表达 QA evidence，Tekon 会窄归一化为 schema 字段；缺失状态、含糊状态或只有 criterion 而无证据字段时仍会失败。评审类 artifact 必须额外写入合法 reviewScope、reviewProcess、decision 和 findings 数组；如果有 finding，每项必须包含合法 severity 和 message，不能用 reviewRole、reviewedArtifacts 或数组/对象形式的 reviewScope 替代这些 schema 字段；无效 finding ownerRole 会被保留到 message，不会放宽核心 review 字段。真实 provider 默认总超时为 1 小时，无 stdout/stderr 或受控输出目录文件进展默认 15 分钟超时；CLI run 可用 --timeout-ms、--no-progress-timeout-ms、--progress-heartbeat-ms 显式覆盖，Web dashboard 也提供对应输入项，适合把明确长程任务拉到 2 小时或更长。最终配置会写入 provider snapshot 以支持 resume；命令执行会写入 *.progress.json，记录状态、最近输出时间、stdout/stderr 字节数、受控输出目录文件数量和字节数、最近输出目录活动时间、elapsed、总超时、无进展超时、timeoutReason 和 heartbeat 次数。CommandGateway 会排除自身 stdout/stderr/progress 文件，只把 artifact、manifest 等产物文件变化计入进展；diff 级续期和可恢复 job runner 仍是后续增强。若 Codex 超时或非零退出但已写完有效 manifest，adapter 会在必需 artifact 校验通过后把该节点视为完成并继续进入 gate；缺少 workflow 必需 artifact、artifact schema 不合法、path/symlink 边界失败，或非 timeout signal 终止时，该节点会失败。QA validation 会记录 tested ref，QA signoff、PR package 和 readiness 会校验所测对象与交付对象一致。Codex provider 不会自动创建 PR、merge 或上线，远端副作用仍由 delivery create-pr --approve-human 控制；该命令中的受控 git/gh 命令及前置只读 probe 也会写入 progress JSON，并使用同一 1 小时总超时和 15 分钟无 stdout/stderr 或受控输出目录文件进展超时。

4.6 查看结果

tekon status
tekon review
提示 PR/CI 证据尚未写回前，eval readiness 通常会因为 pr-prepared、pr-created 或 remote-ci-passed 失败而保持 ready=false。

4.7 准备 PR 材料

tekon delivery prepare

这一步当前只支持 standard-delivery 治理 run，只生成本地 PR 包，不 push、不创建 PR。生成前会执行 pre-PR readiness：workflow 必须 passed、无 pending human gate、验证 gate 与安全扫描满足、AC evidence 完整、QA release signoff 必须通过且绑定 QA validation 记录的 tested ref。未满足时不会生成 PR 包。

4.8 创建远端 PR

tekon delivery create-pr --approve-human
注意 这一步会产生真实远端副作用：push 分支并调用 GitHub CLI 创建 PR。执行前会重新生成并校验 PR 包，因此不会绕过 pre-PR readiness、QA signoff 和所测即所得校验。受控 git/gh 命令默认 1 小时总超时、15 分钟无 stdout/stderr 或受控输出目录文件进展超时，并写入 progress JSON；delivery 分支名和 base branch 会拒绝 --mirror、:branch、空白、..、@{ 等不安全 ref。

4.9 查询远端 CI

tekon delivery ci-status

如果希望等待 checks 到终态：

tekon delivery ci-watch --max-attempts 20 --interval-ms 15000

远端 CI 证据写回后，再执行：

tekon eval readiness

4.10 默认上下文规则

天工的常规 CLI 使用方式是"进入目标仓库根目录后执行短命令"。默认推断规则如下：

Repo

优先使用 --repo；不传时从当前目录向上查找 .tekon/config.yaml，找不到时使用当前 Git 仓库根目录。

Demand shape

demand shape 默认写入 .tekon/demands/；demand approve 默认批准最近需求卡，如果最近需求卡已经批准，历史未批准需求卡必须显式传 --shape <path>；eval demand-shape 默认评估最近一张需求卡。

Run

run 没有需求文本且没有 --demand-file 时，默认读取最近需求卡，且该需求卡必须已批准；status、review、eval readiness、delivery prepare 等默认使用最近一次 run。

Human decision

approval summary、eval approval-summary、approval reject 和 resume --approve-human 默认使用最近的 pending human decision；如果同一 run 同时存在多个 pending decision，必须显式传 --decision-id。

显式传参的三种情况 1. 从其它目录操作目标仓库时传 --repo；2. 查看或处理历史对象时传 --run-id、--shape、--demand-file、--decision-id；3. 执行高风险动作时保留 --approve-human 或 --allow-dirty-base 作为明确人工确认。

如果你显式传了 --repo、--run-id 或 --decision-id 查看跨仓库或历史对象，review 和 approval summary 会输出带 id 和 repo 的精确后续命令，避免复制短命令后误操作到最新 run、最新待审批项或当前 shell 目录。

SECTION 05

核心概念

理解天工的十一个基础构件。

5.1 Repo

目标仓库，即你希望天工处理的项目。天工自身仓库和目标仓库可以相同，也可以不同。

5.2 .tekon/

天工在目标仓库中的运行态目录。保存配置、SQLite 数据库、run artifact、gate 日志、worktree、demand shape 文件和 Web session token。通常不提交。

5.3 Run

一次 workflow 执行。每个 run 有唯一 runId（如 run_xxx）。查看状态、审阅、准备 PR 都围绕 run id。

5.4 Workflow

一组有顺序和依赖的角色节点。内置模板：standard-feature、bugfix、test-improvement、docs-update、plan-only、standard-delivery。

5.5 Role

执行节点的角色，例如 PM、RD、QA、Reviewer、PMO。角色决定 prompt、知识和工具策略。

5.6 Gate

验证关卡：build、lint、test、schema、security-scan、human、independent-review、role-scope、ac-evidence、qa-signoff、process-completeness。Gate 不通过时 workflow 不应被当成可交付。

5.7 Artifact

结构化产物：需求卡、代码变更说明、测试报告、审阅报告、PR 包、CI 状态。是人工审阅和自动评估的主要证据。

5.8 Provider

执行节点的 agent 后端。当前可选 mock、claude-code 和 codex；真实 provider 需要本机认证并写入 manifest。

5.9 Review Surface

聚合审阅面。CLI 命令是 review，Web dashboard 也使用同一套数据。把用户最需要看的东西放在一起。

5.10 Readiness

单次 run 的工作就绪度评估。回答："这次 run 的 workflow、gate、artifact、PR 准备、真实 PR 和远端 CI 证据是否已经完整？"

5.11 Work Usability

样本集级评估。回答："天工是否已在足够多真实样本上表现稳定，可以作为受控工作工具试用？"

Workflow 执行流程概览

需求 → demand shape → 需求卡 → approve ↓ workflow select ↓ ┌──────────────────── run ────────────────────┐ │ │ │ PM → RD → QA → Reviewer → PMO │ │ ↓ ↓ ↓ ↓ ↓ │ │ gate gate gate gate gate │ │ │ └─────────────────────────────────────────────┘ ↓ review → delivery prepare → create-pr → ci-watch → eval readiness

SECTION 06

命令详解

天工 CLI 全部命令的参数、用法和故障处理。命令均假设在目标仓库根目录执行。

6.1 init 初始化目标仓库 
tekon init
常用参数
  • --repo <path> — 从其它目录初始化指定仓库时使用。不传时自动使用当前 Git 仓库根目录或当前目录。
结果
  • 创建 .tekon/、config.yaml、repo-profile.yaml、web-session.json
  • 创建运行所需目录
问题处理

如果目标目录不是 Git 仓库，涉及 diff/worktree/delivery 的功能可能失败。如果 web-session.json 被删除，可重新执行 init。

6.2 workflow preflight 运行前检查命令画像 
tekon workflow preflight
常用参数
  • 第一个位置参数 — 模板名；不传时默认 standard-delivery。
  • --repo <path> — 只在跨仓库检查时使用。
判断结果
  • resolved — 可执行
  • missing — 缺命令，需要补 repo profile
  • not-applicable — 用户已显式声明不适用
  • not-command-gate — 语义 gate，不需要 repo profile 命令
注意 不要为了通过 gate 随意配置 notApplicable。security-scan 不应通过 notApplicable 绕过。
6.3 workflow select 根据需求推荐模板 
tekon workflow select "补齐 CLI 单元测试"
说明

这不是动态规划，不会自动保存 workflow。人可以覆盖推荐，但建议用 eval workflow-selection 检查。

6.4 demand shape 需求塑形成需求卡 
tekon demand shape "需求文本"
常用参数
  • --no-write — 只预览，不写入 .tekon/demands/
  • --repo <path> — 只在跨仓库塑形时使用
  • --format json — 输出 JSON
结果
  • JSON 源文件
  • Markdown 审阅稿
  • 推荐模板
  • 风险和验收信息
问题处理

如果 openQuestions 不为空，建议先补充需求。如果推荐模板不符合预期，用 workflow select 和 eval workflow-selection 核对。

6.5 demand approve 人工批准需求卡 
tekon demand approve
常用参数
  • 位置参数或 --shape <path> — 指定需求卡 JSON 路径；不传时默认批准最近需求卡。如果最近需求卡已经批准，历史未批准需求卡必须显式指定。
  • --actor <name> — 记录批准操作者
注意 批准需求卡不等于批准 PR 创建，也不绕过后续 gate。
6.6 run 发起一次 workflow 
# 模板运行
tekon run "需求文本" --template standard-delivery --agent mock

# 需求卡运行（默认读取最近已批准需求卡）
tekon run

# 使用本机 Codex CLI provider
tekon run --agent codex

# 动态 dry-run
tekon run --dynamic --dry-run "需求文本" --agent mock
常用参数
  • --template <name> — 使用内置模板
  • --demand-file <path> — 使用指定已批准需求卡；不传需求文本时默认读取最近需求卡并要求它已批准
  • --agent mock — 使用 mock provider
  • --agent claude-code — 使用 Claude Code adapter
  • --agent codex — 使用本机 Codex CLI adapter；要求 codex 在 PATH 中且已完成本机认证
  • --allow-dirty-base — 允许基于未提交改动运行
  • --repo <path> — 只在跨仓库运行时使用
判断结果

输出 runId 后，用 status 和 review 继续检查。status=passed 不代表可以自动合入。有 pending human gate 时，需要先处理审批。

6.7 status 查看 run 当前状态 
tekon status
常见字段
  • workflowStatus — 整体状态
  • currentNodeId — 当前节点
  • artifacts — 产物数量
  • gates — gate 数量
  • pendingHumanDecisions — 待人工决策数量
6.8 approval summary 生成可复制审批摘要 
tekon approval summary
常用参数
  • --run-id <runId> — 查看指定历史 run 的审批项时使用；不传时默认最近的 pending human decision
  • --decision-id <decisionId> — 同一 run 有多个 pending decision 或需要指定历史决策时使用
  • --max-chars <n> — 限制 artifact 和日志预览长度，默认 1200
  • --json — 输出结构化 JSON
摘要包含

decision id、run id、node id、需求标题、风险、exact command、影响文件状态、readiness 失败项、证据入口、批准/拒绝命令、Web 处理入口。

注意 摘要是审批材料，不是通知机器人。默认拒绝命令不携带操作者示例，避免复制错误审计信息。
6.9 eval approval-summary 检查审批摘要完整性 
tekon eval approval-summary
检查项
  • pending decision 是否存在
  • 风险信息、exact command、影响信息
  • 批准/拒绝入口是否可复制
  • 证据上下文是否存在
  • 正文是否包含关键命令

ready=true — 摘要具备基本审批材料。ready=false — 不建议拿给 reviewer 决策，应先补证据。

6.10 resume --approve-human 批准 human gate 并继续 
tekon resume --approve-human
注意事项
  • 只在你已审阅风险和证据后使用
  • 会按 run 创建时落库的 provider 快照恢复
  • 同一 run 有多个 pending decision 时必须传 --decision-id <decisionId>；显式指定后只批准这一条 decision
  • 旧 run 缺 provider 快照时会拒绝继续，避免从真实 provider 意外切到 mock
6.11 approval reject 拒绝并阻断 workflow 
tekon approval reject
常用参数
  • --run-id <runId> — 拒绝指定 run 的 pending decision 时使用
  • --decision-id <decisionId> — 同一 run 有多个 pending decision 或要精确拒绝某个 decision 时使用
  • --actor <name> — 记录拒绝操作者
  • --note <text> — 记录拒绝原因
  • --repo <path> — 只在跨仓库操作时使用
结果
  • human decision 变为 rejected
  • workflow 阻断，human gate 分类为 human-rejected
  • review 会显示人工拒绝语义，不会误判成命令策略拒绝
6.12 review 查看完整审阅材料 
tekon review
重点章节
Readiness Failed Checks Evidence Navigation Gate Failure Triage Delivery Changed Files Artifacts Gate Logs PR Body PR Package Next Commands
如何使用
  • 先看 readiness 失败项
  • 再看 Evidence Navigation 指向的证据
  • Gate 失败时先看 triage 分类和建议命令
  • 准备 PR 前看 Changed Files 和 PR Body
6.13–6.17 delivery 交付与 CI 查询 
# 只看交付计划，无远端副作用
tekon delivery dry-run

# 生成本地 PR 包
tekon delivery prepare

# 人工批准后创建远端 PR（真实副作用）
tekon delivery create-pr --approve-human

# 只读查询 PR checks
tekon delivery ci-status

# 可选：指定 PR 或分支
tekon delivery ci-status --selector <prUrl|branch>

# 轮询 PR checks 到终态
tekon delivery ci-watch --max-attempts 20 --interval-ms 15000
delivery prepare 结果
  • .tekon/runs/<runId>/delivery/pr-package.md
  • .tekon/runs/<runId>/delivery/pr-body.md
  • delivery-package artifact
  • delivery.pr-prepared 审计事件
create-pr 必要条件
  • 已安装并认证 gh
  • 目标远端有创建 PR 权限
  • 主工作区除 .tekon 外没有未提交改动
  • 用户明确传入 --approve-human
  • workflow 已 passed，AC evidence、安全扫描和 QA release signoff 已满足，且 QA signoff 绑定 QA validation tested ref。
  • 长程 push、gh pr create 或 create-pr 前置只读 probe 会写入 command progress JSON；默认 1 小时总超时、15 分钟无 stdout/stderr 或受控输出目录文件进展超时。
ci-status 结果
  • 写入 ci-status artifact
  • 记录 delivery.ci.checked 审计事件
  • 后续 PR 包和 readiness 可看到远端 CI 证据
ci-watch 注意 不 rerun CI、不 merge、不上线。
6.18–6.19 eval 就绪度与可用性评估

SECTION 08

如何判断结果是否可信

不要只看"命令退出 0"。按顺序检查这八项。

1. status → workflow 是否 passed 2. review → 失败项和证据是否能解释 3. Changed Files → 影响文件是否符合预期 4. Artifacts → 需求/变更/测试/审阅证据是否完整 5. Gate Logs → build/lint/test/security 是否真的跑过 6. PR Package → PR body 是否能让 reviewer 看懂 7. CI Status → 远端 checks 是否已记录 8. eval readiness → PR/CI 证据是否完整
核心原则 如果其中任何一步说不清楚，不要继续创建 PR 或批准高风险动作。

逐项说明

1. status — workflow 是否 passed

status=passed 表示所有节点完成，但不代表可以自动合入。仍需检查 gate 和 artifact。

2. eval readiness — 是否 ready

ready=true 表示 PR 准备、真实 PR 和远端 CI 证据也已满足 required 项；PR/CI 前 ready=false 是预期状态。

3. review — 失败项和证据

重点看 Readiness Failed Checks、Evidence Navigation 和 Gate Failure Triage。

4. Changed Files — 影响文件

确认影响范围符合预期，没有意外文件被修改。

5. Artifacts — 证据完整性

检查需求卡、变更说明、测试报告、审阅报告是否齐全。

6. Gate Logs — 验证记录

确认 build、lint、test、security scan 确实执行并通过。

7. PR Package — PR 材料

PR body 应该能让 reviewer 理解改动内容和原因。

8. CI Status — 远端检查

如果已创建 PR，确认远端 checks 已记录。

SECTION 09

常见问题处理

九类常见故障的诊断和处理方式。

9.1 workflow preflight 显示 missing command

原因：目标仓库没有配置对应命令。

处理：

  1. 看 suggestedCommand 是否合理
  2. 把确认后的命令写入 .tekon/repo-profile.yaml
  3. 再跑 workflow preflight
注意 不要直接跳过 gate，除非该命令确实不适用且你能写出原因。

9.2 run 拒绝 dirty base

原因：目标仓库有未提交业务改动。

处理：先提交、stash 或清理无关改动。如果明确要基于当前改动运行，追加 --allow-dirty-base。

说明 --allow-dirty-base 是人工确认，不是安全证明。

9.3 pending human gate

处理流程：

  1. 执行 approval summary
  2. 执行 eval approval-summary
  3. 如果摘要不完整，先看 review 补证据
  4. 批准：resume --approve-human
  5. 拒绝：approval reject

9.4 readiness 不通过

常见原因：

workflow 还没 passed pending human gate 验证 gate 失败 PR 准备包不存在 security scan 失败 artifact 缺验收标准 evidence

处理：先看 review 的 failed checks → Evidence Navigation → Gate Failure Triage。

9.5 delivery create-pr 失败

常见原因：没有 gh、gh auth status 失败、远端权限不足、主工作区 dirty、pre-PR readiness 不满足、delivery 分支名或 base branch 不安全、PR 已存在。

处理：先修认证和工作区状态。若 PR 已存在，看命令是否恢复了 PR URL。不要直接重复执行高风险命令，先看 .tekon/runs/<runId>/delivery/ 和 review。

9.6 ci-status 查询失败

常见原因：run 没有 PR URL、selector 不对、gh pr checks 不支持目标、远端无 checks、权限不足。

处理：用 --selector <prUrl|branch> 明确指定。先手动确认 gh pr checks 是否可用。对非 GitHub host，当前需要后续 adapter 支持。

9.7 Artifact 被拒绝入库

原因：产物命中了明显密钥或 token 模式。

处理：删除密钥内容，使用安全摘要或脱敏示例，重新运行相关节点或 provider。这只是基础扫描，不等于完整 DLP。

9.8 Web 写操作被拒绝

常见原因：session token 错误、.tekon/web-session.json 不存在、项目 root 不在允许范围。

处理：重新执行 init 生成 token。确认 Web 启动时的 TEKON_PROJECT_ROOT 正确。不要提交 token。

9.9 Codex provider 运行失败

常见原因：本机未安装 codex、Codex CLI 未完成认证、provider 没有写入必需 artifact manifest，或用户 args 试图覆盖 profile / sandbox / approval / filesystem / config / bypass 参数。

处理：先执行 codex --version 和最小 codex --profile internal --sandbox workspace-write --ask-for-approval on-request exec --help smoke。该 smoke 只验证 CLI 与 internal profile；真实 Tekon run 会在 exec 前受控追加 --add-dir <TEKON_OUTPUT_DIR>，只开放本节点 artifact 输出目录。再查看 .tekon/runs/<runId>/<nodeId>/ 下 stdout/stderr、artifact-manifest.json、字面 TEKON_ARTIFACT_MANIFEST 和 artifact 内容，并确认结构化 JSON 有非空 title 和 body。真实 provider 默认 1 小时总超时和 15 分钟无 stdout/stderr 或受控输出目录文件进展超时；可通过 CLI run 的 --timeout-ms、--no-progress-timeout-ms、--progress-heartbeat-ms 或 Web dashboard 对应字段显式放大外层预算。长程任务仍应结合日志、manifest 和 artifact 变化判断是否有进展。不要把真实 provider 失败降级成 mock 通过。

详细流程见 docs/manual/codex-provider-smoke.md。

SECTION 11

边界与安全原则

天工的设计边界、推荐使用范围和不建议使用范围。

必须明确

不会自动 merge 不会自动上线 不会自动扩大权限 不是生产级 OS 沙箱 不是完整 DLP 没有远程多租户权限模型 只写回 CI 查询，不控制 CI Codex provider 是本机 CLI 集成 飞书 IM 通知尚未接入

推荐使用范围

✓ 内部工具

内部工具的研发和维护。

✓ 研发效能工具

提升研发效率的工具开发。

✓ 测试补齐

补充单测、回归测试、API 测试。

✓ 文档补齐

用户手册、API 文档、技术方案。

✓ 低风险 bugfix

逻辑分支修复、状态处理调整。

✓ 中小需求

可回滚、可人工审阅的改动。

不建议当前使用范围

生产数据写入 支付 / 权限 / 合规 / 隐私 自动发布 无人审阅的远端操作 长时间真实 LLM 且缺恢复预案

手册更新规则

后续每次功能、行为、CLI/Web 入口、参数、错误处理、边界或用户流程发生变化后，都必须评估是否需要更新本手册。

需要更新的典型情况：

  • 新增或删除 CLI 命令
  • 参数语义变化
  • Web dashboard 新增写操作
  • Gate、readiness、work-usability 规则变化
  • 运行目录、artifact、PR 包或审计结构变化
  • 新增真实 provider 支持
  • 安全边界、人工审批边界或远端副作用规则变化
  • 用户常见故障处理方式变化

如果一次迭代判断不需要更新本手册，应在最终回复或提交说明中说明理由。
