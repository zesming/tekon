Donkey：面向技术基建团队的 AI 自动交付系统产品方案

核心结论

Donkey 建议定位为面向技术基建团队的 AI 自动交付系统。长期方向是让 B/D 类需求自动推进到可验收 PR；当前本地版本先聚焦目标阶段判断、需求/技术方案草案、可配置 Coding Agent Adapter、本地分支与 commit、本地测试验收、风险报告和交付证据包，不自动 push 或创建 PR。

一、背景

技术基建团队通常既是内部产品的需求方，也是研发、测试、验收和流程推进方。内部平台、研发效能工具、任务系统、诊断工具、自动化能力等需求，本质上有明确的工程价值，但经常被大量流程性工作拖慢。

AI 编码、Agent 编排、工具协议和自动化测试能力正在快速成熟，但行业经验同时说明：单点 AI 工具并不天然带来团队级交付效率提升。Donkey 的机会不是再做一个 Coding Agent，而是做 Agent 交付流程的组织、验证和审计层。

二、目标

Donkey 是面向技术基建团队的 AI 自动交付系统。完整产品形态下，用户提出需求目标后，Donkey 自动完成需求澄清、技术方案、任务拆解、任务委派、开发、测试、验收报告和 PR 提交。当前本地版本覆盖目标阶段判断、文档草案、本地 Coding Agent Adapter、本地分支与 commit、本地测试验收、风险降级和证据归档。

一句话定位：让技术基建团队从“自己提出需求、自己拆、自己做、自己测、自己催”转向“提出目标，验收交付”。

三、用户路径

  1. 用户输入需求想法。
  2. Donkey 自动理解目标与上下文，生成需求卡和验收标准。
  3. Donkey 匹配仓库画像和 SOP，生成技术方案与任务拆解。
  4. 当前本地版本由 Donkey Runner 先创建本地分支，再调用配置的 Coding Agent Adapter 写代码，运行本地测试验收，通过后创建本地 commit；后续阶段接入 CI/E2E 和稳定自动修复。
  5. Review Agent 输出风险摘要。
  6. 当前 MVP 生成交付证据包；后续阶段创建 PR 和交付证据包。
  7. 用户最终验收。

四、思路

核心原则：Autonomy-first, Risk-gated。

长期原则是执行阶段尽量全自动、交付阶段强证据验收；低中风险需求后续默认跑到 PR，高危动作动态插入人工确认。当前本地版本的自动化边界先到文档草案、本地代码修改、本地分支与 commit、本地测试验收、风险报告和证据包。

核心模块：

六、效果验证评估标准体系

长期北极星指标：标准技术基建需求从输入到可验收 PR 的自动闭环率。

当前本地版本验证指标先看目标阶段判断、代码修改、本地分支与 commit、本地测试验收和风险降级是否可靠。

维度指标本地版本达标线
效率需求输入到本地代码变更、commit 和测试验收报告耗时可在本地自动生成代码变更报告、Git 提交报告和证据包
自动化目标阶段判断准确率内置 EvalCase 100%，后续扩真实样本

体验最终验收决策效率80%+ 用户能在 5 分钟内判断接受或退回
