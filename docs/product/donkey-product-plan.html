Donkey：面向技术基建团队的 AI 自动交付系统产品方案

核心结论

Donkey 建议定位为面向技术基建团队的 AI 自动交付系统。第一阶段不做通用项目管理平台，也不追求自动上线，而是聚焦 B/D 类需求：研发效能工具功能迭代与 Bugfix/小优化，默认自动跑到可验收 PR，并以交付证据包帮助人只判断结果是否符合预期。

一、背景

技术基建团队通常既是内部产品的需求方，也是研发、测试、验收和流程推进方。内部平台、研发效能工具、任务系统、诊断工具、自动化能力等需求，本质上有明确的工程价值，但经常被大量流程性工作拖慢。

AI 编码、Agent 编排、工具协议和自动化测试能力正在快速成熟，但行业经验同时说明：单点 AI 工具并不天然带来团队级交付效率提升。Donkey 的机会不是再做一个 Coding Agent，而是做 Agent 交付流程的组织、验证和审计层。

二、目标

Donkey 是面向技术基建团队的 AI 自动交付系统。用户提出需求目标，Donkey 自动完成需求澄清、技术方案、任务拆解、任务委派、开发、测试、验收报告和 PR 提交。

一句话定位：让技术基建团队从“自己提出需求、自己拆、自己做、自己测、自己催”转向“提出目标，验收交付”。

三、用户路径

  1. 用户输入需求想法。
  2. Donkey 自动理解目标与上下文，生成需求卡和验收标准。
  3. Donkey 匹配仓库画像和 SOP，生成技术方案与任务拆解。
  4. Agent 自动开发、运行 CI/单测/E2E，并进行自动修复。
  5. Review Agent 输出风险摘要。
  6. Donkey 创建 PR 和交付证据包。
  7. 用户最终验收。

四、思路

核心原则：Autonomy-first, Risk-gated。

执行阶段尽量全自动，交付阶段强证据验收；低中风险需求默认跑到 PR，高危动作动态插入人工确认。

五、方案

核心模块：

模块产品职责关键产物
交互层承接需求输入、进度查看、最终验收和高危确认需求入口、验收页、通知
需求池把自然语言需求转成结构化需求卡目标、范围、非目标、验收标准
仓库画像沉淀每个试点仓库的 SOP、测试方式、风险边界和历史经验项目画像、验证规则、危险动作清单
流程编排中心根据需求和仓库画像决定流程、角色、任务和回退策略执行计划、任务委派、状态流转
Role Agents承担需求、方案、开发、测试、审查和报告等角色任务方案、代码变更、测试结果、审查建议
产物与风险门禁管理关键产物和风险判断需求卡、任务、测试报告、风险说明
交付证据包把最终交付转化为可验收、可审查、可追溯的结果PR 链接、验收结论、证据和风险
