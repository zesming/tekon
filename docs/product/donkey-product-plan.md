# Donkey：面向技术基建团队的 AI 自动交付系统产品方案

## 核心结论

Donkey 建议定位为面向技术基建团队的 AI 自动交付系统。长期方向是让 B/D 类需求自动推进到可验收 PR；当前 0.1.0 MVP 先聚焦目标阶段判断、需求/技术方案草案、本地测试验收、风险报告和交付证据包，不声明已经具备自动开发和自动 PR 交付能力。

## 一、背景

技术基建团队通常既是内部产品的需求方，也是研发、测试、验收和流程推进方。内部平台、研发效能工具、任务系统、诊断工具、自动化能力等需求，本质上有明确的工程价值，但经常被大量流程性工作拖慢。

AI 编码、Agent 编排、工具协议和自动化测试能力正在快速成熟，但行业经验同时说明：单点 AI 工具并不天然带来团队级交付效率提升。Donkey 的机会不是再做一个 Coding Agent，而是做 Agent 交付流程的组织、验证和审计层。

## 二、目标

Donkey 是面向技术基建团队的 AI 自动交付系统。完整产品形态下，用户提出需求目标后，Donkey 自动完成需求澄清、技术方案、任务拆解、任务委派、开发、测试、验收报告和 PR 提交。当前 0.1.0 MVP 只覆盖其中的目标阶段判断、文档草案、本地测试验收、风险降级和证据归档。

一句话定位：让技术基建团队从“自己提出需求、自己拆、自己做、自己测、自己催”转向“提出目标，验收交付”。

## 三、用户路径

1. 用户输入需求想法。
2. Donkey 自动理解目标与上下文，生成需求卡和验收标准。
3. Donkey 匹配仓库画像和 SOP，生成技术方案与任务拆解。
4. 当前 MVP 可运行本地测试验收；后续阶段接入 Agent 自动开发、CI/E2E 和自动修复。
5. Review Agent 输出风险摘要。
6. 当前 MVP 生成交付证据包；后续阶段创建 PR 和交付证据包。
7. 用户最终验收。

## 四、思路

核心原则：Autonomy-first, Risk-gated。

长期原则是执行阶段尽量全自动、交付阶段强证据验收；低中风险需求后续默认跑到 PR，高危动作动态插入人工确认。当前 0.1.0 MVP 的自动化边界先到文档草案、本地测试验收、风险报告和证据包。

## 五、方案

核心模块：

| 模块 | 产品职责 | 关键产物 |
|-|-|-|
| 交互层 | 承接需求输入、进度查看、最终验收和高危确认 | 需求入口、验收页、通知 |
| 需求池 | 把自然语言需求转成结构化需求卡 | 目标、范围、非目标、验收标准 |
| 仓库画像 | 沉淀每个试点仓库的 SOP、测试方式、风险边界和历史经验 | 项目画像、验证规则、危险动作清单 |
| 流程编排中心 | 根据需求和仓库画像决定流程、角色、任务和回退策略 | 执行计划、任务委派、状态流转 |
| Role Agents | 当前承担需求、方案、测试、审查和报告；Phase 2+ 扩展到开发和自修复 | 当前：方案、测试结果、审查建议；Phase 2+：代码变更、修复记录 |
| 产物与风险门禁 | 管理关键产物和风险判断 | 需求卡、任务、测试报告、风险说明 |
| 交付证据包 | 把最终交付转化为可验收、可审查、可追溯的结果 | 当前：验收结论、证据和风险；后续：PR 链接 |

## 六、效果验证评估标准体系

长期北极星指标：标准技术基建需求从输入到可验收 PR 的自动闭环率。

当前 0.1.0 MVP 验证指标先看目标阶段判断、测试验收和风险降级是否可靠。

| 维度 | 指标 | 0.1.0 MVP 达标线 |
|-|-|-|
| 效率 | 已有方案到测试验收报告耗时 | 可在本地自动生成证据包 |
| 自动化 | 目标阶段判断准确率 | 内置 EvalCase 100%，后续扩真实样本 |
| 质量 | 验收证据可读性 | HTML 证据包能支持人工判断 |
| 质量 | 测试命令执行可信度 | 记录命令、退出码、日志路径和结论 |
| 风险 | 高危动作误执行次数 | 0 |
| 体验 | 最终验收决策效率 | 80%+ 用户能在 5 分钟内判断接受或退回 |

## 七、分步迭代

| 阶段 | 目标 |
|-|-|
| Phase 0 | 试点准备，建立仓库画像、SOP 与风险清单 |
| Phase 1 | 0.1.0 MVP，目标阶段判断、文档草案、本地测试验收、风险报告和交付证据包 |
| Phase 2 | 自动开发、PR 交付、多 Workflow 与自修复增强 |
| Phase 3 | 团队级协同，支持更多仓库、Dashboard 与复用 |
| Phase 4 | 组织级 AI 研发效能平台 |
