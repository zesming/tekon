Donkey MVP 验收报告

日期：2026-06-04

验收对象：Donkey 本地自动开发与验收 MVP

结论：验收通过。当前版本已把一句话输入、目标阶段判断、可配置 Coding Agent Adapter、本地分支与 commit、本地代码修改、本地测试验收、风险降级、TUI 和 HTML 证据包串成可运行闭环；不会自动 push、创建 PR、合入或上线。

实现范围

能力验收结果说明
CLI / 短命令通过支持 init、run、ask、tui、status、show、eval、adapter；npm run d -- "<需求>" 可一句话发起。
TUI通过npm start 可进入交互菜单，支持发起运行、查看最近运行、配置命令和运行内置评测。
文件态状态通过每次运行写入 .donkey/runs/<runId>/state.json，包含 schema version、Intent、Workflow、ToolRun、Evidence 和事件摘要。
Intent / Target Stage Gate通过支持想法、需求、技术方案、任务/变更、PR 和高危需求降级；“按方案执行”进入开发链路，“直接执行测试验收”进入验收链路。
Workflow 裁剪通过已有技术方案可跳过需求梳理；开发目标会显示并执行开发与测试验收；只验收目标不会声明已开发。
Coding Agent Adapter通过通过 commands.develop 调用 Codex、Claude 或自定义本地命令修改工作区；本地分支、测试验收和 commit 由 Donkey Runner 统一执行。
Git 分支与 commit通过开发链路会先检查工作区干净状态，创建 donkey/<runId> 分支；测试通过后提交本地 commit。脏工作区、已 staged 的 .donkey 产物会阻断；Adapter / 测试命令内的常规 commit 和 push 有临时 Git wrapper 与 hook guardrail；Runner-owned git 写操作使用 Donkey 控制的 hooksPath，避免仓库 hook 触发 push/PR；绕过 commit hook 导致 HEAD 或分支变化时会触发不变量阻断。
Tool Gateway工程验证通过统一执行命令，默认拒绝未 allowlist 的命令，拦截 shell 控制符和重定向，以 shell: false 执行 argv；记录 stdout/stderr、耗时、退出码并做敏感信息脱敏。
Evidence / HTML通过运行报告与 Eval 报告均生成 HTML 人审版。
donkey-eval工程验证通过内置 6 个 EvalCase replay，通过率 100%，输入类型准确率 100%，目标阶段准确率 100%，高危误放行 0。当前样本仍需在真实 B/D 需求中继续扩充。

边界说明

  • 当前版本不是正式 Server、Web Console、CI Runner、Container Runner 或团队级任务平台。
  • Donkey 自身不内置编码模型服务；代码修改通过本地 Codex、Claude 或自定义命令等 Coding Agent Adapter 完成。
  • Donkey 会自动创建本地分支与 commit，但不会 push、创建 PR、合入、上线或修改生产高危资源。
  • 当前本地 Runner 不是安全沙箱；对未信任或恶意 Adapter 的硬隔离需要后续容器、无网络或系统级 sandbox Runner。
  • 内置 Intent/Gate 仍是规则化 MVP，后续需要用真实 B/D 类需求样本扩展 EvalCase。
  • 测试通过只代表配置的验证命令通过，最终是否接受交付仍由人审阅 HTML 证据包和本地 diff 决定。
