# Tekon 审批摘要增量报告

日期：2026-06-08

范围：补齐真实工作可用清单中的“审批摘要和通知入口”低风险版本。当前目标是先让 CLI/Web 生成可复制审批材料，不接入飞书 IM，不扩大自动化范围。

## 1. 背景判断

上一份真实工作可用清单指出，Tekon 已经能在 Web 本地处理 human gate，也能通过 CLI `resume --approve-human` 继续运行，但日常工作中 reviewer 仍需要快速看到“为什么要批、风险是什么、会执行什么、影响哪些文件、有哪些证据、如何拒绝”。因此本轮先补可复制审批摘要，后续再决定是否接飞书 IM。

## 2. 已完成能力

| 能力 | 实现位置 | 说明 |
| --- | --- | --- |
| 审批摘要聚合 | `packages/core/src/approval/summary.ts` | 从 pending human decision、review surface、readiness、diff 和 evidence links 生成可复制摘要。 |
| 摘要评估 | `evaluateHumanApprovalSummary`、CLI `eval approval-summary` | 检查摘要是否具备 pending decision、风险、命令、影响、证据、批准入口、拒绝入口和正文。 |
| CLI 审批入口 | `packages/cli/src/index.ts` | 新增 `approval summary` 输出摘要，新增 `approval reject` 将 pending decision 标记为 rejected，并以 `human-rejected` gate 分类阻断 workflow。 |
| Web 展示 | `packages/web/src/server/api/root.ts`、`packages/web/src/client/App.tsx` | `gate.list` 返回同一审批摘要，dashboard 待审批区展示只读摘要文本；Web reject 与 CLI 使用同一 `human-rejected` gate 语义。 |
| 文档同步 | `README.md`、`CHANGELOG.md`、`docs/manual/tekon-v2-user-manual.*`、工作可用报告 | 记录新命令、新评估和“飞书 IM 尚未接入”的边界。 |

## 3. 验证覆盖

- Core 单测：`approval/summary.test.ts` 覆盖摘要生成、评估和拒绝命令不含不可执行示例值；`gate/human-gate.test.ts` 覆盖拒绝后 workflow 阻断和 `human-rejected` 分类；`review/surface.test.ts` 覆盖人工拒绝不会误判为命令策略拒绝。
- CLI 单测：`run-cli.test.ts` 覆盖 `approval summary`、`eval approval-summary`、`approval reject` 和拒绝后的 review surface。
- Web API：`write-auth.test.ts` 覆盖 pending decision 返回审批摘要和评估结果，以及 Web reject 后的 `human-rejected` gate/review surface。
- Web e2e：`release-dashboard.test.ts` 覆盖 dashboard 展示审批摘要和拒绝入口文本。
- 类型检查：`pnpm typecheck` 已验证新增导出、CLI/Web 消费和 React 类型。

## 4. 边界

- 当前新增的是“可复制审批摘要”，不是飞书 IM 通知机器人。
- `approval summary` 只读取目标仓库内 review surface 已允许读取的证据，不执行 push、PR 创建、merge 或上线。
- `approval reject` 只处理 pending human decision，会阻断 workflow，不会自动修复或重跑；审批摘要里的默认拒绝命令可直接复制执行，如需记录操作者可额外传入真实账号或姓名。
- 摘要中的真实影响文件依赖 delivery diff 是否可用；不可用时会明确写出 unavailable 和原因。

## 5. Reviewer 结论

第一轮最高思考 reviewer 结论为 CHANGES_REQUESTED，必须修复项包括：拒绝命令包含不可直接执行的操作者示例值；人工拒绝复用 `rejected` 分类会被 review surface 误判为命令策略拒绝，且 CLI/Web 拒绝后的 gate 上下文不一致。

已修复摘要：审批摘要默认拒绝命令改为不带 `--actor` 的可复制命令；摘要评估增加可复制命令结构和值检查；CLI/Web 拒绝统一写入 `human-rejected` gate 分类；review surface 为 `human-rejected` 输出人工拒绝语义；补充 core、CLI、Web API 和 review surface 回归断言。

复查过程：后续多轮最高思考 reviewer 继续收紧摘要评估边界，指出 `rejectCommand` 不能依赖 Web 入口兜底、必须校验 `--decision-id`、`--run-id` 和 `--repo` 的有效值，并且不能接受 `--actor` 或 `--actor=value` 形式的操作者示例。上述问题已全部修复并补充负向用例。最终复查结论：必须修复项为无；建议项为后续可将 `rejectHumanGate` 的 pending invariant 下沉到 core API，并继续用真实审批样本验证摘要体验。

## 6. 后续建议

1. 用真实 run 样本验证摘要是否足够 reviewer 在飞书消息中判断批准/拒绝。
2. 对真实审批样本统计缺失字段，反向补充 risk label、impact files 和 evidence links。
3. 摘要稳定后再接飞书 IM，只做通知和链接，不绕过 human gate。
