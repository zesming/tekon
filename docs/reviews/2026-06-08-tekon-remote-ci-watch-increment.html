# Tekon 远端 CI watch 增量报告

日期：2026-06-08

范围：补齐 PR 创建后等待远端 PR checks 进入终态的第一版能力。本文只覆盖只读轮询、退避等待、每次查询落库和审计记录；不覆盖自动 rerun CI、自动 merge、自动上线或非 GitHub provider。

## 1. 背景判断

飞书方案和本地缺口分析都把近期目标收敛为“受控推进到可审 PR，并让人能基于证据快速判断能不能接受”。上一轮 `delivery ci-status` 已能单次读取 GitHub PR checks 并把结果写回 evidence，但真实工作里 PR 创建后经常会遇到 checks 仍在 pending 的状态。

如果 Tekon 只能单次查询，人需要手动重复执行命令或打开 GitHub 页面刷新；这会削弱交付证据包的闭环体验。本轮补齐 `delivery ci-watch`，让 Tekon 在只读边界内等待 checks 进入 `passed`、`failed` 或 `skipped` 终态，并保留每次查询证据。

## 2. 已完成能力

| 能力              | 实现位置                                                                                                                                          | 说明                                                                                                                                             |
| ----------------- | ------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| CI watch core API | `packages/core/src/delivery/ci-status.ts`                                                                                                         | 新增 `watchPullRequestCiStatus`，循环调用 `queryPullRequestCiStatus`，支持 `maxAttempts`、`intervalMs`、`backoffMultiplier` 和测试注入 `sleep`。 |
| 终态判断          | `packages/core/src/delivery/ci-status.ts`                                                                                                         | `passed`、`failed`、`skipped` 视为终态；`pending`、`unknown` 会继续轮询直到达到次数上限。                                                        |
| 每次尝试留证      | `packages/core/src/delivery/ci-status.ts`                                                                                                         | 每次轮询仍写入独立 `ci-status` artifact 和 `delivery.ci.checked` 审计事件，避免只保留最后一行结果。                                              |
| watch 完成审计    | `packages/core/src/delivery/ci-status.ts`                                                                                                         | 结束时写入 `delivery.ci.watch-completed`，payload 包含 selector、status、attempts、maxAttempts、terminal 和 artifactId。                         |
| CLI 命令          | `packages/cli/src/index.ts`                                                                                                                       | 新增 `delivery ci-watch --run-id <runId> [--selector <prUrl\|branch>] [--max-attempts N] [--interval-ms MS] [--backoff N]`。                     |
| 单元和 e2e 覆盖   | `packages/core/__tests__/delivery/ci-status.test.ts`、`packages/cli/__tests__/run-cli.test.ts`、`packages/cli/__tests__/e2e/release-flow.test.ts` | 覆盖 pending 到 passed、达到 max attempts 后返回非终态、CLI 输出和 release fixture 下的 fake `gh pr checks`。                                    |

## 3. 当前边界

- `delivery ci-watch` 只等待和记录 checks 状态，不会 rerun CI。
- `delivery ci-watch` 不会 push、不创建 PR、不 merge、不上线。
- 当前仍依赖 GitHub CLI 的 `gh pr checks`；GitLab、内部 Git 平台、Buildkite、Jenkins 等需要后续 provider adapter。
- 当前本地测试使用 fake `gh` fixture；真实 GitHub PR 上仍需记录 PR URL、checks 状态变化、artifact id 和审计事件。
- `remote-ci-passed` 在 readiness 中仍是 recommended，不是 required；团队可以先审本地 PR 包，再决定是否创建远端 PR 或等待 CI。

新增覆盖点：

- `watchPullRequestCiStatus` 在第一次 pending、第二次 passed 时返回 `terminal=true`，记录 2 个 `ci-status` artifact，并调用一次 sleep。
- `watchPullRequestCiStatus` 在持续 pending 且达到 `maxAttempts` 时返回 `terminal=false`，并记录 `delivery.ci.watch-completed`。
- CLI `delivery ci-watch` 在 fake `gh` 下输出 `ciStatus=passed`、`terminal=true`、`attempts`、`maxAttempts`、`checks`、`artifactId` 和 selector。
- release e2e 在受控 fake `gh pr checks` 下覆盖 `delivery create-pr` 后执行 `delivery ci-watch`。

## 5. 后续仍需

1. 在受控 GitHub 测试仓库创建真实 PR 后执行 `delivery ci-watch`，记录状态从 pending 到终态的样本。
2. 明确无 checks、required checks 长时间 pending、`gh` 未认证和仓库无权限时的用户提示分类。
3. 如果内部主力代码平台不是 GitHub，增加 CI status provider adapter，而不是继续扩大 `gh` 路径。

## 6. Reviewer 结论

最高思考 reviewer 复查结论：APPROVED。必须修复项为无。

Reviewer 建议后续补强两项：一是给 watch 退避后的派生 delay 增加 cap 或总超时，避免极端参数下 timeout 过大；二是可补一条“先 passed、再 ci-watch max attempts pending 后 readiness=false”的直接回归。两项均不阻塞本次增量交付。
