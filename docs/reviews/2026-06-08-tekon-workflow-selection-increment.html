# Tekon 受控 Workflow Selection 增量报告

日期：2026-06-08

范围：补齐 P1 “Workflow 选择”。本增量不实现动态 workflow 自动规划，也不让模型直接生成可执行 workflow；它只在受控内置模板之间做确定性推荐、人工可覆盖和可评估的选择。

## 1. 背景判断

真实工作中，不同需求不应该都硬套 `standard-feature` 或 `bugfix`。测试补齐、文档更新和仅方案类任务有不同的验收重点和风险边界；如果用户每次都手选模板，Tekon 很容易把“工具接入成本”转嫁给使用者。

本轮选择最小可验证方案：增加一组受控模板，并让需求塑形和 CLI 选择器给出推荐与原因。动态规划仍停留在 dry-run 或建议阶段，不进入自动执行。

## 2. 已完成能力

| 能力             | 实现位置                                                                 | 说明                                                                                         |
| ---------------- | ------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------- |
| 受控模板集合     | `workflows/test-improvement.yaml`、`workflows/docs-update.yaml`、`workflows/plan-only.yaml` | 在 `standard-feature` 和 `bugfix` 之外，新增测试补齐、文档更新和仅方案模板。                 |
| 需求到模板推荐   | `packages/core/src/demand/shape.ts`                                      | `shapeDemand` 会推荐 `bugfix`、`test-improvement`、`docs-update`、`plan-only` 或 `standard-feature`。 |

| CLI 入口         | `packages/cli/src/index.ts`                                              | 新增 `workflow select`，可基于原始需求或 demand shape 输出推荐模板、候选模板和原因。          |
| Web 入口         | `packages/web/src/client/App.tsx`                                        | Web 模板选择器展示新增受控模板；需求塑形后会把推荐模板填入运行模板。                         |

## 3. 当前边界

- `workflow select` 是确定性启发式，不是 PM LLM 动态规划。
- `plan-only` 不产出 `code-changes`，包含 reviewer human gate；它适合方案和评审，不等价于可交付 PR。
- `test-improvement` 会要求 test/e2e evidence；目标仓库缺少 e2e 命令时，应在 repo profile 中补命令或显式 `notApplicable`。
- `docs-update` 仍保留 build/lint/security-scan gate；文档仓库不适用的普通命令需要显式配置，不会被静默跳过。
- 人仍可选择不同模板，但 `eval workflow-selection` 会把与推荐不一致的选择暴露出来。

## 4. 验证记录

阶段性验证已通过：

```bash
npm exec --yes -- pnpm@10.12.1 --filter @tekon/core test:unit -- --run packages/core/__tests__/demand/shape.test.ts packages/core/__tests__/workflow/template.test.ts
npm exec --yes -- pnpm@10.12.1 --filter @tekon/cli test:unit -- --run packages/cli/__tests__/run-cli.test.ts
npm exec --yes -- pnpm@10.12.1 --filter @tekon/cli test:e2e
npm exec --yes -- pnpm@10.12.1 typecheck
```

新增覆盖点：

- Core：`test-improvement`、`docs-update`、`plan-only` 推荐；错误模板选择评估失败；所有受控内置模板可被解析。
- CLI：`workflow select` 输出推荐模板，`eval workflow-selection` 对错误模板返回 `ready=false` 和失败项；release e2e 覆盖选择与评估命令。
- Web e2e：dashboard 的模板选择器展示 `test-improvement`、`docs-update` 和 `plan-only`。

## 5. 后续仍需

1. 动态 workflow 仍只支持 dry-run；后续如果要把动态规划落成模板，需要人工确认后保存为项目 workflow。
2. 受控模板还需要真实仓库样本验证：测试补齐模板的 e2e 命令缺失体验、文档仓库的 `notApplicable` 配置、plan-only 的人工评审体验。
3. Web 目前只展示可选模板；后续可以把 `workflow select` 的推荐原因和评估结果直接展示到 dashboard。

## 6. Reviewer 结论

最高思考 reviewer 复查结论：APPROVED，必须修复项为无。

复查确认：新增三个内置模板可解析；`test-improvement` 和 `docs-update` 的 code-producing node 保留 build、lint、security-scan 和 schema gate；`plan-only` 不产出 `code-changes`，文档也明确它不等价于可交付 PR。`shapeDemand` 的分类顺序能避免“功能需求 + 要求 e2e 通过”被直接误判成测试类，同时可识别明确测试补齐需求。`workflow select` / `eval workflow-selection` 只做受控模板推荐与评估，未知模板会进入 failed check，没有扩大自动规划、自动执行或 gate 绕过边界。

非阻塞建议：后续可给内置模板增加更强断言测试，显式锁定 code-producing node 必须包含 security-scan/schema；也可统一 `workflow select --shape` 与 `eval workflow-selection --shape` 的相对路径解析体验。
