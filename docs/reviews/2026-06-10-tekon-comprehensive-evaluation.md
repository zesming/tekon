### 2.2 工程质量亮点

#### 2.2.1 确定性工作流引擎

核心设计原则：**编排层绝不调用 LLM**。工作流的选择、调度、状态转换全部由确定性状态机完成，LLM 仅在 Agent 执行节点时被调用。

```
需求输入 → 需求塑形（LLM）→ 工作流选择（确定性）→ 节点调度（确定性）
  → Agent 执行（LLM）→ 门禁检查（确定性）→ 状态转换（确定性）→ ...
```

**优势**：可预测、可审计、无调度层幻觉风险。相比 DeerFlow 2.0 使用 LLM 做任务分解，Tekon 的方式更适合需要合规和审计的场景。

#### 2.2.2 哈希链审计日志

每个审计事件包含前一个事件的哈希值，形成防篡改链：

```
event[0] → hash(event[0])
event[1] → hash(event[1] + hash(event[0]))
event[2] → hash(event[2] + hash(event[1] + hash(event[0])))
```

任何篡改都可被检测，满足审计合规要求。

#### 2.2.3 Git Worktree 隔离

每个执行节点获得独立的 git worktree 租约，Agent 的代码修改在隔离分支上进行，不会污染主工作区。worktree 生命周期由系统管理（创建 → 使用 → 清理）。

#### 2.2.4 门禁自动修复循环

```
Agent 执行 → 门禁检查 → 失败 → 生成修复节点 → Agent 重新执行 → 门禁再检查
                                    ↓（连续失败达上限）
                               升级到人工确认
```

7 类门禁：build / lint / test / e2e-pass / security-scan / schema / human

#### 2.2.5 密钥扫描

共享密钥扫描器被三个模块复用：
- **security-scan 门禁**：扫描代码变更
- **artifact store**：扫描产物内容
- **command gateway**：扫描命令输出

覆盖私钥、OpenAI 风格密钥、AWS 访问密钥、常见 token 模式。

### 2.3 工程质量短板

| 问题 | 当前状态 | 风险 | 建议 |
|------|---------|------|------|
| **单文件 CLI** | 2,360 行全在 `index.ts` | 随功能增加，文件膨胀、职责不清、merge 冲突 | 按命令域拆分（demand/run/delivery/admin），保持单入口但内部分模块 |
| **Provider 单一** | 仅 Claude Code adapter + mock | 供应商锁定，Claude Code 价格/能力变化直接影响系统 | 优先接入 Trae Agent（字节自研 SOTA） |
| **Web Dashboard 基础** | 基础 React SPA，CRUD 为主 | 无法支撑项目驾驶舱、审计回放等高级用户场景 | 按文档 4 定义的 Project Cockpit 视图重构 |
| **无并行执行** | 工作流节点仅顺序执行 | 影响效率，无法满足文档中"架构设计与 UI 设计并行"的场景 | 引擎支持 DAG 调度 |
| **缺乏真实验证** | 无真实需求端到端交付数据 | 无法证明核心价值——自动闭环率、效率提升 | 立即启动 10 个真实需求 MVP 验证 |

## 四、能力方向合理性评估

### 4.1 战略方向判断：✅ 高度合理

Tekon 的核心定位——**"受控 AI 软件交付"（Governed AI Software Delivery）**——精准踩中了 2026 年行业从"AI 编码工具"向"AI 交付治理"转型的关键节点。

| 方向选择 | 行业验证信号 | 评价 |
|---------|------------|------|
| **"受控交付"而非"自动编码"** | DeployFlow 定义 Governed AI Engineering 范式；GitHub Copilot 增加 Agent HQ 管控层；Devin 强调"像管理工程师一样管理 Agent" | ✅ 差异化明确，避开与 Claude Code/Cursor 的直接竞争 |
| **确定性调度器** | DeerFlow/Copilot 用 LLM 编排，Tekon 用确定性状态机。行业白皮书（Atos、Rattix）推荐风险分级自治 | ✅ 更可预测、可审计，适合合规场景 |
| **产物驱动而非聊天驱动** | SDD（Spec-Driven Development）社区共识：结构化规格说明降低 50% 错误率；arXiv 论文验证 | ✅ 与最佳实践完全一致 |
| **聚焦技术基建团队** | 内部工具类需求标准化程度高、验收标准明确、风险可控——最适合 AI 自动交付验证 | ✅ 切入场景精准 |
| **Generation/Review 分离** | SDD 社区共识：产出者 ≠ 评审者；METR 研究证实 AI 使用不当反降效率 19% | ✅ 角色设计核心原则 |
| **风险分级自治** | 76% DevOps 团队已集成 AI 到 CI/CD；DORA 报告强调小批量、强验证 | ✅ 与安全/DevOps 最佳实践一致 |
| **不自动合入/不自动上线** | Veracode 报告：AI 生成代码安全漏洞率为人工的 2.74 倍 | ✅ 保守但正确的安全边界 |

### 4.2 竞品定位分析

Tekon 不与任何单一竞品直接竞争，而是占据一个独特的交叉位置：

```
                    AI 编码工具                   项目管理工具
                ┌──────────────────┐        ┌──────────────────┐
                │ Claude Code      │        │ Jira / Linear    │
                │ Cursor / Copilot │        │ Backstage        │
                │ Codex / Trae     │        │ Harness          │
                └────────┬─────────┘        └────────┬─────────┘
                         │                           │
                         │    Tekon 的位置            │
                         │    ↓                      │
                    ┌────┴───────────────────────┴────┐
                    │  受控 AI 软件交付编排层           │
                    │  (Governed AI Delivery)          │
                    │                                  │
                    │  消费编码 Agent 作为 Provider     │
                    │  输出交付证据包 + 风险门禁        │
                    └─────────────────────────────────┘
```

| 竞品类型 | 代表产品 | 与 Tekon 的关系 |
|---------|---------|----------------|
| AI 编码 Agent | Claude Code, Cursor, Codex, Trae | Tekon **消费**它们作为 Provider，不竞争 |
| 自主 Agent | Devin, Replit Agent, Copilot Workspace | Tekon 在端到端交付上竞争，但有更强的门禁和审计 |
| 多 Agent 框架 | CrewAI, LangGraph, AutoGen | 基础设施层，Tekon 是自研引擎而非使用框架 |
| 内部开发平台 | Backstage, Harness, Humanitec | IDP 聚焦基础设施自助，不涉及 AI 交付工作流 |
| Agent 管理 | Multica, Factory Droids | 最接近竞品，但 Tekon 有更完整的工作流编排和门禁 |

### 4.4 方向风险预警

| 风险 | 严重度 | 说明 | 建议 |
|------|--------|------|------|
| **愿景过大，资源分散** | 🔴 高 | 5 份文档描绘了从 MVP 到"AI 产研组织操作系统"的宏大愿景，但当前实现仅覆盖 ~35% | 严格聚焦 Beta-G 定义的 MVP 范围，不提前投入 Phase 2+ |

## 五、后续优先方向建议

### 5.1 P0：立即执行（0-4 周）

#### 5.1.1 真实需求 MVP 验证

**理由**：所有文档（尤其是 Doc 2）都强调"用真实需求证明价值"，Doc 2 明确定义了 6 项 MVP 达标指标。当前 0 项验证。

**行动**：
1. 选择 2-3 个技术基建仓库作为试点
2. 录入 10 个真实 B/D 类需求（Bugfix、小优化、工具功能迭代）
3. 跑完整工作流：需求塑形 → 方案 → 开发 → 测试 → PR
4. 收集数据：

| 指标 | MVP 达标线（Doc 2） | 当前值 |
|------|-------------------|--------|
| 需求输入到 PR 创建耗时 | 比人工下降 30%+ | ❓ 未度量 |
| 无中途人工介入到 PR 比例 | 50%+ | ❓ 未度量 |
| 验收标准可判定率 | 90%+ | ❓ 未度量 |
| PR 一次审查通过率 | 60%+ | ❓ 未度量 |
| 高危动作误执行次数 | 0 | ❓ 未度量 |
| 最终验收决策效率（5 分钟内判断） | 80%+ | ❓ 未度量 |

**产出**：MVP 验证报告，含每项指标的达标/未达标分析和优化建议。

#### 5.1.2 接入 Trae Agent 作为第二 Provider

**理由**：
- 字节 SE Lab 出品，SWE-bench Verified SOTA
- 解除 Claude Code 供应商锁定
- 为内部推广增加说服力（"字节自研 Agent 驱动"）
- 验证 Provider 抽象协议的通用性

**行动**：
1. 研读 Trae Agent 的 CLI 接口和产物协议
2. 实现 `TraeAgentAdapter implements AgentAdapter`
3. 用同一需求分别跑 Claude Code 和 Trae Agent，对比结果
4. 在 `runtime/` 层增加 provider 选择配置

#### 5.1.3 交付证据包信息层级重构

**理由**：Doc 2 明确定义了交付证据包的 5 层信息架构，这是用户**唯一真正看到的核心界面**。当前 review surface 未按此层级组织。

**目标层级**（来自 Doc 2）：

```
第 1 层：结果层    — 是否完成、是否建议接受、失败项和高风险项
第 2 层：失败与风险 — 默认优先展示失败项和高风险项
第 3 层：证据层    — 验收标准逐条通过情况 + CI/测试/日志/截图
第 4 层：审查层    — Diff 摘要 + Reviewer 关注点 + 风险分析
第 5 层：PR 入口   — 查看变更、接受或退回
```

**行动**：重构 `review/surface.ts`，按上述层级组织输出。

### 5.2 P1：短期执行（1-2 月）

#### 5.2.1 工作流并行执行

**理由**：当前引擎仅支持顺序执行节点。Doc 3/4/5 均提到并行场景（架构设计与 UI 设计并行、多 RD Agent 并行开发）。

**行动**：
- 引擎支持 DAG 调度：分析节点间依赖关系，无依赖节点并行启动
- 并发限制：最大并行 Agent 数可配置
- 并行结果聚合：所有并行节点完成后统一进入 Gate

#### 5.2.2 Web 驾驶舱升级

**理由**：Doc 4 定义了完整的 Project Cockpit 视图，当前 Dashboard 仅有基础 CRUD。

**目标视图**：
- 项目当前阶段 + 流程进度图（可视化节点状态）
- 当前责任角色 + Agent 实时状态
- 阻塞与风险一览
- 产物状态 + Gate 通过情况
- 待人类确认事项（审批队列）
- 工具执行摘要（耗时、成本、成功/失败）

#### 5.2.3 SWE-bench 评测

**理由**：外部调研发现，行业对 AI 交付系统的质量判断越来越依赖可比较的基准测试数据。

**行动**：
- 用 Tekon 工作流跑 SWE-bench Verified 子集（50-100 个 issue）
- 对比纯 Agent（Claude Code / Trae Agent 单独跑）vs Tekon 编排后的效果
- 发布评测报告，证明"编排 + 门禁"带来的质量提升

#### 5.2.4 多 Provider 抽象增强

**理由**：Adapter 接口已有，但需要标准化和文档化以支持更多 Agent。

**行动**：
- 统一 Provider 接口规范（输入/输出/错误/超时/重试）
- 实现 OpenAI Codex adapter
- 支持运行时 Provider 切换（同一工作流可在不同 Provider 间 fallback）

### 5.3 P2：中期执行（2-4 月）

#### 5.3.1 技能沉淀体系

**理由**：所有文档（尤其 Doc 4/5）都强调"技能是长期竞争力"和"组织经验复用"。
