Tekon Web UI 最新复审报告

审查日期：2026-06-12

审查对象：用户最新一轮修复后的 packages/web Web UI、HTTP/RPC/API、 安全边界、E2E 覆盖，以及与 docs/technical/tekon-web-architecture.md 的符合度。

1. 总体结论

当前版本基础验证全部通过，也补上了多项上一轮缺口：redaction 直接文本修复、 delivery.dryRun / delivery.ciStatus / progress.list contract、output parse、Create PR 确认 E2E、 Sec-Fetch-Site 和生产 HTML fallback。

但按技术方案 MVP 验收口径，当前仍 不能认定通过。本轮未发现 P0，但仍有 4 个 P1 必须修复。

2. 验证结果

命令 结果 说明
pnpm --filter @tekon/web typecheck 通过 TypeScript 编译检查通过。

3. 已改善项

  • redactTextPreview 直接文本脱敏已有改善，并新增 redaction 单测。
  • delivery.dryRun、delivery.ciStatus、progress.list 已进入 RPC contract。
  • dispatchApiCall 已返回 spec.output.parse(result)，roleItemSchema 已移除 systemPrompt。
  • 新增 Create PR 确认弹窗 E2E。
  • HTTP 增加 Sec-Fetch-Site 校验，生产路径开始服务 dist/index.html。

4. P1 必须修复项

P1 展示 DTO 的 server-side redaction 仍不完整

redactTextPreview 直接文本已改善，但 redactObject 对普通字符串仍原样返回；artifact.summary、审批 note、 readiness/evidence、nextCommands、gate triage 等展示文本没有统一递归脱敏。

影响：artifact.list、gate.list、audit.list、 review.get 仍可能泄露 token、secret、Bearer 值。

P1 delivery.dryRun / delivery.ciStatus 语义未达到 MVP

delivery.dryRun 调用 createPullRequestPreparation()， 会写 delivery-package artifact、pr-package.md、pr-body.md 和 audit，不是无副作用 dry-run。delivery.ciStatus 当前只读本地状态并伪造 delivery check，不是真实远端 CI。

P1 fake gh Create PR E2E 仍未充分证明显式批准边界

取消弹窗后只断言没有 UI result banner；确认时 waitForResponse 匹配任意 200 /api/rpc，没有过滤 path === 'delivery.createPr'，也没有检查 fake gh.log 在取消、背景关闭、确认前为空。

P1 progress.list 未覆盖架构要求的 progress 摘要

当前只映射 node/status/started/updated/elapsed/timeout/heartbeat， 缺少最近 stdout/stderr 活动、outputDir 活动、接近 no-progress timeout 的风险判断、 timeoutReason 语义化展示和 redacted command 展示。

5. P2 风险项

  • P2 output schema 已 parse，但 Zod object 默认丢弃未知字段，不是 strict drift 检测边界。
  • P2 生产静态资源服务缺少路径边界，且测试依赖 ignored dist。
  • P2 本报告已更新为最新复审结论，交付前仍需确认 Markdown 与 HTML 同步。

6. 修复顺序

  1. 补展示 DTO 全链路 redaction，覆盖所有展示字符串和建议命令。
  2. 修 delivery.dryRun 无副作用语义，或调整命名和技术方案。
  3. 修 delivery.ciStatus，调用真实 core CI 查询并写 ci-status artifact。
  4. 加硬 Create PR E2E：检查 fake gh.log、请求体和 PR URL/audit。
  5. 补完整 progress.list 摘要字段和 UI 展示。
  6. 公开 DTO schema 改 strict，并加固生产静态资源路径边界。

7. 最终判断

当前版本具备较好的基础可运行性，验证命令全部通过；但从 Tekon 架构目标看， 剩余问题集中在生产语义和安全边界。因此本轮结论是： P0 未发现，但 MVP 仍不通过；需先修 4 个 P1。
