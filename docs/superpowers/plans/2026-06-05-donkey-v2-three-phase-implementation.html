Donkey V2 三阶段落地实施计划

源稿：docs/superpowers/plans/2026-06-05-donkey-v2-three-phase-implementation.md。本 HTML 为人工审阅版。

目标：在三个阶段内实现 Donkey V2 技术方案中的核心能力：安全可恢复的 Agent 执行内核、角色化 Workflow 编排、动态 Workflow、硬 Gate、Artifact/Audit、CLI/Web 驾驶舱、PR 交付和 dogfooding 验收。

架构：采用 pnpm monorepo：packages/core 提供纯 TypeScript 领域模型、状态机、仓储、执行内核和编排 API；packages/cli 提供本地命令入口和 TUI；packages/web 提供本地只读优先、受控写操作的 Next.js 驾驶舱。

一、三阶段边界

阶段目标可验收结果不允许后移的能力
阶段一：安全可恢复内核 先把执行边界、持久化、worktree、Gate、Artifact/Audit 打实 mock Agent 跑完最小 workflow；中断后可恢复；危险命令被拒绝；human gate 会暂停 Tool Gateway、WorktreeManager、状态持久化、GateResult、Audit hash chain
阶段二：角色化 Workflow 产品闭环 实现角色文件系统、模板 workflow、动态 workflow、约束注入、完整 CLI donkey run 可用模板或动态 spec 执行到本地证据包 Dynamic dry-run、Constraint mutation、角色技能/知识注入、autoFix repair node
阶段三：交付与可观察产品面 实现 PR 交付、Web 驾驶舱、人工确认、指标、手册、dogfooding 能从需求到 PR URL 和证据包；Web 可审阅项目、产物、Gate 和审计 SCM Delivery、Web human approval、指标沉淀、HTML 用户手册

二、外部资料依据

资料资料内容对 Donkey 的判断依据
Git worktree 官方文档 一个 Git 仓库可支持多个 working tree，每个 worktree 是带独立元数据的工作目录。 并行 Agent 必须真实创建独立 worktree，不能把原仓库路径直接传给所有 Agent。
Node.js child_process 官方文档 spawn 管道容量有限，需要持续消费 stdout/stderr；同步 child process 会阻塞事件循环。 Agent Runner 和 Gate Runner 必须流式消费输出，Gate 不使用 execSync(commandString)。
SQLite WAL 官方文档 WAL 支持读写并发，但仍要按 SQLite 的写入锁模型设计事务。 Donkey 需要单写者队列、busy timeout、短事务和恢复索引。
Claude Code permissions 文档 Agent 可通过 permission modes、hooks、allow/deny 规则控制工具使用。 tools.yaml 不能只是 prompt 文本，必须编译成 provider 权限配置或外层 gateway 规则。
OpenAI Codex approvals/security Codex 安全运行依赖 sandbox、approval 和网络访问边界组合。 AgentAdapter 合约必须显式表达 sandbox/approval 能力。
pnpm workspace 文档 pnpm-workspace.yaml 是 workspace 包发现的根配置。 V2 采用 pnpm workspace 管理 core/cli/web，根 lockfile 固化依赖。
tRPC Next.js App Router 文档 tRPC 在 App Router 下通过初始化 router/procedure 暴露端到端类型 API。 Web 只通过 core API 和 tRPC 读写 SQLite 状态，避免复制业务逻辑。
GitHub CLI PR 文档 gh pr create 可从当前分支创建 PR，成功后输出 PR URL。 PR 交付必须有显式 SCM Delivery 模块、认证检查和失败恢复。

三、阶段一：安全可恢复内核

阶段目标：实现不依赖真实 LLM 的安全执行内核。完成后，mock Agent 可以在独立 worktree 中产出 artifact，状态写入 SQLite，Gate 会真实执行，human gate 会暂停，审计日志可追溯。

pnpm install
pnpm --filter @donkey/core test -- --run
pnpm --filter @donkey/core build
pnpm --filter @donkey/core test:e2e -- --run
Task 1: Monorepo and Test Harness
创建 root/package/core 基础结构、Vitest workspace、core smoke export。验收：core 测试和 build 通过。
Task 2: Domain Types and Runtime Config
定义 Demand、Project、WorkflowInstance、Node、Artifact、Gate、Audit、AgentAdapterConfig、CommandPolicy、WorktreeLease，并用 Zod 校验外部 YAML/JSON 输入。
Task 3: SQLite Persistence, Migrations, and Single Writer Queue
实现 WAL、foreign keys、busy timeout、schema migrations、repositories、write queue、recoverable run 查询。
Task 4: Artifact Store and Append-Only Audit Logger
实现 artifact 版本化文件存储、schema 校验、摘要注入、audit hash chain 和篡改检测。
Task 5: CommandGateway and Safe Process Runner
使用 argv command model，不执行 shell 字符串；实现 allow/deny/human approval、cwd scope、timeout、process group kill 和日志流。
Task 6: WorktreeManager
所有 git worktree add/remove/prune 必须通过 CommandGateway argv 命令执行；使用稳定路径和分支命名，拒绝脏 base，禁止删除 .donkey/worktrees/ 外路径。
Task 7: AgentAdapter Interface and Claude/Mock Runners
实现 mock adapter 与 Claude Code adapter；默认不使用 bypass permissions；支持 stdin/arg/file prompt mode、流式输出和 timeout；无法证明 sandbox/approval/permission 映射的真实 adapter 必须拒绝执行。
Task 8: Gate Engine and HumanGate
实现 build/test/lint/e2e/schema/security/human Gate，持久化 GateResult，human gate 暂停与恢复，autoFix 以 repair node 表达。
Phase 1 Exit Gate
  • core 测试全部通过。
  • temp repo E2E 可以创建 run、租赁 worktree、执行 mock Agent、保存 artifact、运行 schema gate、写 audit、清理 worktree。
  • 危险命令在 spawn 前被拒绝。
  • human gate fixture 可以暂停并通过 repository API 恢复。
  • 审阅记录保存到 docs/reviews/。

四、阶段二：角色化 Workflow 产品闭环

阶段目标：在阶段一内核上实现技术方案中的角色系统、Workflow 模板、动态 Workflow、约束系统和核心 CLI。完成后，用户可以用模板或动态模式运行一个需求到本地交付证据包。

pnpm --filter @donkey/core test -- --run
pnpm --filter @donkey/cli test -- --run
pnpm --filter @donkey/cli test:e2e -- --run
pnpm build
Task 9: Role File System
实现项目级、用户级、内置角色优先级；加载 skills/tools/knowledge；把 tools.yaml 编译成 CommandGateway policy 和 provider permission config。
Task 10: Workflow Templates and State Machine
实现 YAML parser、input/output shorthand、状态机、standard-feature 和 bugfix 模板；拒绝缺少 review/validation/gate 的不合规 workflow。
Task 11: Constraint Validator and Workflow Mutation
实现硬约束、条件约束、软建议；高风险注入 human gate，安全类注入 security review 和 security-scan，数据类要求 rollback-plan。
Task 12: Workflow Engine Scheduler
实现持久化 run 创建、phase/node 调度、artifact dependency resolution、gate/repair loop、并行节点安全检查和中断恢复。
Task 13: Dynamic Workflow Mode
定义 PM 生成的 WorkflowSpecDraft JSON schema，支持 dynamic dry-run、constraint mutation preview 和 save-as template。
Task 14: CLI Core Commands
实现 init/run/status/pause/resume/cancel/role/workflow/constraints/log/clean，支持 mock 与 Claude adapter。
Task 15: Real Agent Smoke and Template Product Loop
跑模板 mock 全流程、dynamic dry-run、高风险 constraint 注入；可选 Claude smoke 只在临时 worktree 和受控权限下运行。
Phase 2 Exit Gate
  • donkey run --template standard-feature --agent mock 完成本地证据包。
  • donkey run --dynamic --dry-run --agent mock 展示约束后的 workflow preview。
  • pause/resume/cancel/status/log/clean 基于持久化状态工作。
  • Constraint validator 阻断不安全动态 workflow。
  • 审阅记录保存到 docs/reviews/。

五、阶段三：交付、Web 驾驶舱和 Dogfooding

阶段目标：补齐技术方案中的 PR 交付、Web Dashboard、人工确认界面、效果评估、用户手册和 Donkey-on-Donkey 自举验收。

pnpm test -- --run
pnpm build
pnpm --filter @donkey/web test:e2e -- --run
pnpm --filter @donkey/cli test:e2e -- --run
Task 16: SCM Delivery and PR Creation
实现 SCM status、delivery package、git commit/push、gh pr create、dry-run 和 human approval。
Task 17: Web API and Project Context
实现 tRPC routers；Web 必须读取显式 DONKEY_PROJECT_ROOT 或 CLI 配置，不能默认假设 .。
Task 18: Web Dashboard UI
实现项目概览、artifacts、audit、gates、roles、workflows、settings、人类审批界面；Playwright 截图验证桌面和移动宽度。
Task 19: Metrics, Evaluation, and Dogfooding Reports
抽取耗时、自动化比例、Gate 通过率、重试、人类介入、高危动作等指标；生成 Markdown/HTML dogfooding report。
Task 20: Documentation and Manual
生成用户手册 Markdown/HTML，更新 README、CHANGELOG，并检查正式文档没有未完成标记。
Task 21: Release Packaging and Final Acceptance
全量 build/test/E2E/coverage，生成最终验收报告 Markdown/HTML。
Phase 3 Exit Gate
  • CLI 可以创建本地证据包和 PR dry-run；真实 PR 创建必须有人类确认或受控 fixture。
  • Web dashboard 显示项目、产物、Gate、审计、角色、Workflow、设置和 human approvals。
  • 用户手册 HTML 与当前 CLI 行为一致。
  • Dogfooding 和最终验收报告同时存在 Markdown 与 HTML。

六、范围决策

覆盖能力：TypeScript monorepo、角色文件夹、Workflow 模板、动态 Workflow、约束系统、确定性 Orchestrator、Artifact Store、Schema Gate、Gate Engine、Human Gate、Audit Logger、Git worktree、SQLite 恢复、CLI、Web、PR 交付和 dogfooding。

仍然不做：不自动合入 PR，不自动上线；不做远程多租户服务；不做飞书 IM 通知集成；不做多人协作权限模型；Codex 通过 AgentAdapter 和自定义 command 协议可接入，但正式 Codex preset 需单独验收 sandbox/approval 映射。

实现替代与降级：Schema Gate 的第一版使用 Zod schema 作为统一校验层，AJV/JSON Schema 兼容导出后续补；真实 PR 创建依赖 GitHub CLI 认证和远程权限，无认证或高风险仓库必须降级为 --delivery dry-run；文档中的 <date> 是未来审阅文件命名模板变量。
