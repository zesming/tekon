Tekon Codex 自举与后续优先级路线图

本方案将 2026-06-10 两份评估报告、用户修正意见和当前实现状态收敛为一条可执行路线： 先完成 Codex 自举真实 PR 闭环，再用真实证据决定 Trae、证据包、Web、Provider 协议、技能沉淀和 benchmark 的推进顺序。

核心结论

Tekon 的战略方向是合理的：它不应该再做一个 Coding Agent，而应该做受控 AI 软件交付的编排、门禁、审计和证据层。

需要调整的是优先级。P0 不应先做 Trae，也不应先扩大到 10 个外部样本；更稳妥的路径是先把 Codex 作为真实 provider 接入， 再让 Tekon 用自己的真实需求创建一个真实 PR，拿到第一条可信闭环证据。

报告原判断 本方案判断 调整依据
P0 跑 10 个真实需求 P0 先跑 Tekon 自身 1 个真实需求闭环 当前缺真实 provider 和真实 PR 自举证据，先扩大样本会放大噪声
Trae Agent 优先接入 先 Codex，后 Trae 用户已明确先适配 Codex；Codex CLI 非交互模式适合当前 provider 抽象
Web 驾驶舱尽快升级 放到 P2 没有真实 run/PR/CI 数据前，Dashboard 缺少验证对象
SWE-bench 评测值得推进 放到 P4 前后 先证明 Tekon 能处理自身真实需求，再做外部 benchmark

P0：Codex 自举真实 PR

P0 的目标不是“adapter 能跑”，而是“Tekon 能用 Codex provider 处理 Tekon 自己的真实需求，并通过 Tekon 的交付链路创建真实 PR”。

建议需求 补齐 Codex provider 在 CLI/Web/eval/docs 中的可用性，并生成 Codex provider smoke 使用说明。
关键证据 run id、provider snapshot、required artifacts、gate、readiness、PR URL、CI 状态。
完成边界 真实 PR 创建才算闭环；delivery prepare、mock provider、dry-run 均不能替代。

P0 门禁

  • Codex provider 默认命令使用 codex --profile internal --sandbox workspace-write --ask-for-approval on-request --add-dir <TEKON_OUTPUT_DIR> exec，其中 --add-dir 只由 Tekon 受控追加到 artifact 输出目录。
  • 用户配置不得覆盖 profile、sandbox 和 approval，不得开启 danger-full-access 或 bypass approvals。
  • artifact manifest 必须被写出并进入 artifact store；缺失 required artifact 时必须失败。
  • 必须通过 delivery create-pr --approve-human 创建真实 PR。
  • CI 成功或失败都要如实记录，不把失败包装成成功。
当前工作区已有未提交的 Codex core adapter 初稿，这只表示 P0 已开始；CLI/Web 接线、文档、完整验证和真实 PR 证据完成前，P0 仍未闭环。

P1：把一次成功变成可重复能力

P1-A：Codex provider 产品化

CLI、Web、eval、docs、snapshot resume 都要把 Codex 当作一等 provider。失败 run 必须能给出明确诊断，不能只显示 agent 自述。

P1-B：交付证据包 V2

证据包按五层组织：结果层、风险层、证据层、审查层、PR 决策层。默认突出失败项和高风险项，让 reviewer 能在 5 分钟内判断接收或退回。

P1-C：Tekon 自身样本扩容

从 1 个自举样本扩展到 5-10 个 Tekon 自身真实需求，统计耗时、人工介入、artifact 完整率、gate 一次通过率、PR 一次审查通过率。

P2：Provider 和体验层扩展

Trae provider

Trae 是重要方向，但应在 Codex contract 稳定后接入，用同一个 Tekon 需求做 Codex/Trae 对比，输出成功率、artifact 完整率、耗时、人工介入和失败可诊断性。

Web Cockpit V2

Web 首屏回答：run 到哪一步、哪个 gate 失败、provider 是谁、PR/CI/evidence 状态是什么、下一步谁处理。

Provider 协议稳定化

固化 prompt 输入、artifact manifest、provider snapshot、错误模型、安全边界和 resume 语义，避免每接一个 provider 都改 workflow engine。

P3/P4：长线能力

方向 前置条件 产出
并行 DAG 执行 顺序 workflow 在真实样本中稳定 静态 DAG、并发上限、聚合 gate、失败恢复
技能沉淀与反技能 真实 run 数据足够 高评分样本提炼为 skill，重复失败沉淀为 anti-skill
成本和效率计量 Provider usage 或运行时长可采集 按 run/provider/role 统计成本、耗时和人工介入
SWE-bench 和 Backstage 内部真实样本已证明核心价值 外部 benchmark 报告和平台入口

最终建议

未来 4-6 周的里程碑应定为：Tekon 能连续处理 5 个 Tekon 自身真实需求，其中至少 3 个由 Codex provider 创建真实 PR，所有样本都有 work-usability、readiness、gate、review surface 和 CI 状态证据。

这个目标比“接更多 provider”或“做更大 Dashboard”更能证明 Tekon 的核心价值。达到后再推进 Trae、Web Cockpit、provider contract、技能沉淀和 benchmark，优先级就会由真实数据驱动，而不是由愿景文档驱动。
