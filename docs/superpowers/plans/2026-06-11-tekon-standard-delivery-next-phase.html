Tekon Standard Delivery 下一阶段迭代方案

日期：2026-06-11｜状态：已根据 P1-0 自举失败样本和当前代码约束重新调整

目标：先让 Tekon 以 Codex provider 支持可运行的标准交付种子流程，再逐步补齐独立评审、角色边界强校验、AC evidence、QA final signoff、PMO checkpoint 和长程任务可观测能力。

当前事实

  • P0 已完成真实 Codex 闭环：run_d2350140-b1b7-4fca-b01b-e28daac61e31 创建真实 PR #2，CI 通过；Tekon 未自动 merge，PR #2 后续已由人工路径合入 main。
  • P1-0 seed run run_04b37267-2686-42c6-a0a4-9b37410f65f7 在 RD 节点 300 秒超时后中断，PM 已产出 demand-card 和 PRD。
  • 当前 artifact 和 gate 都是闭集 schema，首版模板不能直接引用未实现的 independent-review、role-scope、qa-signoff 或 ac-evidence。
  • 当前 artifactTypeSchema 只支持 demand-card、prd、tech-design、code-changes、test-report、review-report、security-report、rollback-plan、delivery-package、ci-status。
  • 当前 gateTypeSchema 只支持 build、test、lint、e2e-pass、schema、security-scan、human。
  • 因此，independent-review、role-scope、qa-signoff、ac-evidence、process-completeness 等 gate，以及 test-plan、qa-release-signoff、process-checkpoint 等 artifact 不能直接写入首版模板。

调整原则

  • P1-A 首版模板必须 parser-compatible，只使用当前已注册 artifact 和 gate。
  • 角色边界先进入 roles/*/system.md，强制防越权放到后续 deterministic gate。
  • 自举任务必须缩小粒度，避免一个 RD node 同时承担模板、角色、schema、gate 和 PR package。
  • 长程任务采用“总超时 + 进展观测 + 无进展超时 + 可恢复证据”，不只靠等待更久。
  • QA final signoff 先以 QA 节点和 test-report/review-report 表达，后续再引入绑定 delivery branch SHA 或 PR head SHA 的专用 schema。
  • PR 创建、merge、release、deploy、force push 保持人工控制。

资料依据

资料 资料内容 对 Tekon 的判断依据
IBM requirements guideline 好需求应清晰、可验证，支撑沟通、设计、计划和工程活动。 PM demand-card 后需要需求质量评审，不能只依赖 schema。
ISTQB CTFL syllabus 测试活动应维护 test basis、testware、测试结果和追踪关系。 QA test plan、QA validation、AC evidence 和 signoff 需要结构化关联。
DORA change approval 高效变更审批应前移到开发过程中的 review，并用自动化检测补充。 Tekon 应做轻量独立评审和自动 gate，不引入重审批委员会。
Scrum Definition of Done Done 表示增量满足质量标准并处于 usable 状态。 QA final signoff 必须绑定确切交付对象，避免“所测非所得”。
Google Engineering Practices Review 关注设计、功能、复杂度、测试和文档。 RD technical review 和 reviewer change review 需要明确 rubric。
Temporal Activity timeouts 长程 activity 可通过总时限、单次执行时限和 heartbeat 检测失败。 Tekon 长程 node 不能只提高总超时，还要记录进展和无进展超时。
Celery worker time limits 任务可以配置 soft/hard time limit，soft limit 给清理留窗口。 Tekon 应区分温和中断、证据收集和最终强制终止。
GitHub Actions limits CI/自动化平台有运行时长、并发和资源限制，需要显式治理。 Tekon 长程任务要进入可观测队列，不能无限占用本地或 CI 资源。

长程任务策略

层级 能力 判断
L1 默认真实 provider 超时提升到 1 小时，并写入 provider snapshot 本轮落地，降低长程 seed 误杀概率；2 小时以上只作为显式配置
L2 观测 stdout/stderr、diff、manifest mtime、artifact 文件数和命令活动 后续必须做，否则长超时只是更久等待
L3 node heartbeat、无进展超时、可中断恢复、后台 job registry P1-D/P2 做，不塞进 P1-A0

推荐运行策略：

  • maxRuntimeMs 默认 1 小时，后续允许 repo/profile 或 run 级覆盖。
  • noProgressTimeoutMs 默认 15 分钟，只要日志、diff、manifest 或 artifact 有变化就续期。
  • status/review surface 展示 lastProgressAt、lastArtifactAt、stdoutBytes、stderrBytes、changedFiles。
  • 超时后先尝试读取 manifest；manifest 完整且必需 artifact 合法时允许进入 gate。
  • 没有 manifest 且无进展时中断，并把 stderr 尾部、worktree diff 和 mtime 写入诊断证据。

主要风险

  • 长超时会降低误杀，但如果没有进展观测，会让卡死任务占用更久。
  • P1-A 的角色边界只是 prompt 层约束，不是确定性强约束。
  • QA final signoff 若要绑定 PR head SHA，需要处理 PR 创建发生在 workflow 之后的问题；P1-C 应先支持 delivery branch SHA，再补 PR 创建后的二次校验。
  • 当前 engine 仍按 phase/node 顺序执行，不应把模板中的多角色评审理解为已具备复杂 DAG 并行。
  • 如果继续用单个 RD node 承担过大 seed 任务，Codex 超时仍可能复现。
