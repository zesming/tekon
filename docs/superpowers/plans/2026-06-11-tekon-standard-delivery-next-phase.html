Tekon Standard Delivery 下一阶段迭代方案

本方案将 Tekon 从“真实 Codex provider 到真实 PR 的 P0 闭环”升级为“有独立评审、角色边界、证据追踪、QA 最终签核和 PMO 全程治理的标准全链路交付流程”。

结论摘要

当前 P0 真实闭环成立，但不是完整治理流程。下一阶段优先做 standard-delivery 模板、独立评审、角色越权 gate、AC evidence traceability、QA final signoff 和 PR package V2。能用 Tekon 自举验证的任务优先用 Tekon 自己跑，但不让尚未实现的 standard-delivery 假装已经可用。

P1-0：自举种子 用现有 docs-update / Codex 闭环跑首个标准流程种子 PR。
P1-A：流程固化 新增标准全链路 workflow 和角色边界。
P1-B：评审硬约束 独立 agent/process/execution 评审，防止自产自测。
P1-C：证据闭环 AC 到 test plan、test report、QA signoff 自动追踪。
P1-D：真实样本 用 3 个 Tekon 自身需求验证标准流程。
P2：体验与观测 Web Cockpit、Artifact Center、审计回放、成本 telemetry。
P3：平台化 DAG 并行、知识技能飞轮、新角色、release/rollback、组织治理。

自举落地策略

标准流程本身也应该被 Tekon dogfood。策略是分层自举：先用现有 docs-update 和 Codex provider 跑低风险种子任务；再用半标准流程验证 schema/gate；最后等 standard-delivery 可用后，用它验证 PR package、Web、Artifact Center 等后续能力。

批次 使用流程 适合任务 目标
Seed-1 现有 docs-update + Codex provider + 真实 PR 模板草案、角色边界文档、模板解析测试、用户手册说明 用已有能力生成第一条 P1 自举证据。
Seed-2 现有 workflow + 新 schema/gate 验证证据 artifact schemas、independent-review、role-scope、qa-signoff、ac-evidence 在新能力尚未进入标准流程前，验证核心治理能力。
Seed-3 新 standard-delivery PR package V2、Web Cockpit、Artifact Center、审计回放、成本 telemetry、Provider 对比 用标准流程验证标准流程之后的能力。

首个自举任务只新增 workflows/standard-delivery.yaml、角色边界和模板测试； 不实现新 gate runner，不改变交付行为。这样能验证流程模板和角色职责两个基础假设，同时避免伪造新 gate 已生效。

资料依据

资料 资料内容 对 Tekon 的判断依据
IBM requirements guideline 好需求必须清晰、可验证，是多方沟通、设计、项目计划和工程活动的基础。 PM demand-card 后必须有需求质量评审，不能只做 schema 校验。
ISTQB CTFL syllabus 测试流程应维护 test basis、testware、测试结果和角色之间的追踪关系。 Tekon 必须把 AC、QA test plan、QA report 和 signoff 自动串起来。
DORA change approval 高效变更审批更适合前移到开发过程中的 peer review，并用自动化检测补充。 Tekon 应做轻量独立评审和自动 gate，不引入笨重审批委员会。
Scrum Definition of Done Done 描述增量满足产品质量标准、处于 usable 状态。 QA final signoff 应绑定 PR head commit，证明所测即所得。
Google Engineering Practices Review 关注 design、functionality、complexity、tests、documentation。 RD technical review 和独立 Reviewer review 应有明确 rubric。

标准流程

1. PMdemand-card + 独立需求内审
2. RD/QA需求接口评审，只评自身工作入口
3. RDimplementation plan + 独立技术评审
4. QAtest plan + 独立测试方案评审
5. PM验证测试方案是否覆盖需求意图
6. RD实现 + build/lint/test/security gates
7. Reviewer独立代码/文档审阅
8. QA在目标 SHA 上验收并签核
9. QA Reviewer复核 final signoff
10. PMO检查证据链和流程完整性
11. Delivery人工批准 create/update PR
12. Human审阅、决定是否合入或发布

Artifact Contract

Artifact Owner 作用
demand-review PM reviewer 判断需求必要性、合理性、范围和 AC 质量。
requirement-interface-review RD / QA 判断需求产物是否足够支撑自身工作。
implementation-plan RD 开发前技术计划、风险、测试策略和回滚。
technical-review RD reviewer 评审技术方案合理性。
test-plan QA 测试验收方案和 AC 覆盖矩阵。
qa-release-signoff QA 绑定目标 SHA 的最终质量签核。
process-checkpoint PMO 过程完整性、证据链、gate、人工决策和交付准备度。

每个 review artifact 必须记录 reviewedArtifactId、producerAgentId、reviewerAgentId、producerExecutionId、reviewerExecutionId 和 contextMode=isolated。

指标与风险

指标 P1 目标
Required artifact completeness 100%，缺失则阻断 delivery。
Independent review enforcement 100%，同 agent 或 execution 自评必须失败。
AC evidence coverage Accepted PR 达到 100%。
QA signoff SHA match 100%，signoff SHA 必须匹配 PR head 或 delivery branch SHA。
Human review time 低风险变更 5 分钟内可判断。

最大风险是流程过重。控制策略是保留轻量 docs-update 模板，把 standard-delivery 用作默认受控交付和中等风险以上需求，不把所有小文档改动强行套重流程。
