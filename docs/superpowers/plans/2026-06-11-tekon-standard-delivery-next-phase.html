2. 流程判断

问题 判断 Tekon 落点
PM 需求卡后是否有 PM 内评和 RD/QA 外评 需要。PM 内审确认必要性、合理性、边界和验收口径；PM 外评确认需求意图对 RD/QA 可交付；RD/QA 外评只评自己接口。 pm-demand-review、pm-requirement-intent-review、rd/qa-requirement-interface-review
RD 开发前是否有技术评审 需要。RD 评设计、风险、改动面、验证和回滚，不评业务必要性。 rd-implementation-plan、rd-technical-review
QA 测试方案是否和 PM 联合验证 需要。QA 评可执行性，PM 评需求意图覆盖，不替 QA 设计测试细节。 qa-test-plan-review、pm-test-plan-intent-review
PMO 是否每个节点关注完整性 需要，但 PMO 不替专业角色评审内容；已写入逐节点 pmo.node-checkpoint 审计事件。 逐节点 pmo.node-checkpoint、末端 pmo-checkpoint
最后交付是否由 QA 做 合理。QA 对“所测即所得”签署 release signoff；PMO 负责流程证据完整性。 qa-release-signoff、delivery-package

角色参与评审必须限制在自身职责范围内，正式评审必须由独立 agent 或独立进程产出 reviewProcess.mode=independent-agent|independent-process，避免自产自测。

3. 标准执行模板

pm-demand-card
-> pm-demand-review
-> pm-requirement-intent-review
-> rd-requirement-interface-review + qa-requirement-interface-review
-> rd-implementation-plan
-> rd-technical-review
-> qa-test-plan
-> qa-test-plan-review + pm-test-plan-intent-review
-> rd-code-change
-> reviewer-change-review
-> qa-validation
-> qa-release-signoff
-> qa-release-signoff-review
-> pmo-checkpoint
  • independent-review gate 拦截 self review 和未批准 review。
  • role-scope gate 拦截角色越权 scope 和 reviewerRole/node role 不一致。
  • 非 code-changes 节点在 worktree finalize 前会被源码变更 guard 拦截，防止 PM/QA/PMO 节点越权修改实现。
  • ac-evidence gate 要求所有验收标准都有 passed evidence。
  • qa-validation 会记录被测 delivery ref；qa-signoff gate 要求 QA signoff 的 targetRef、validatedRef 与最新 qa.validation.ref 一致。
  • delivery prepare 和 delivery create-pr 共享 pre-PR readiness：不要求 PR 已创建或远端 CI 已通过，但要求 workflow passed、无 pending human gate、验证 gate/安全扫描满足、AC evidence 完整、QA signoff 通过且绑定 qa.validation.ref。
  • process-completeness gate 要求 PMO checkpoint 中必需节点已 passed 且无缺失信息。
  • 每个节点通过后都会写入 pmo.node-checkpoint 审计事件，末端 process-completeness gate 再检查整体流程证据。
  • workflow preflight 对无需命令的 schema/semantic gate 输出 status=not-command-gate，只把 repo profile 显式不适用输出为 status=not-applicable。

5. 长程任务方案

层级 能力 本轮状态 判断
L1 真实 provider 和受控 PR 创建命令默认 timeoutMs=3600000，CLI/Web run 支持显式覆盖 已落地 默认 1 小时合理；2 小时以上应通过 run 显式配置。
L2 progress JSON 记录状态、stdout/stderr、输出目录文件指标、elapsed、timeout、heartbeat 已增强 能判断任务是否仍在输出，或是否仍在写入 artifact/manifest。
L3 noProgressTimeoutMs 基于 stdout/stderr 和受控输出目录文件变化续期，默认 15 分钟 已增强 比无限拉大总超时更稳；diff 级续期和外部 job runner 属于后续增强。
L4 process registry、resume token、外部 job runner、节点级 checkpoint 待做 适合 P1-D/P2，避免本地会话断开导致长任务丢失。
L5 队列、资源配额、并发、取消、成本和审计 待做 适合平台化阶段，不放进当前 MVP。

因此，把长程任务总超时拉到 1h/2h 是合理的，但应作为外层预算；是否卡死要靠 heartbeat、stdout/stderr、artifact/manifest 文件变化，以及可取消/可恢复机制来判断。

8. 资料依据

  • IBM requirements guideline：需求应清晰、可验证，支撑沟通、设计、计划和工程活动。
  • ISTQB CTFL syllabus：测试活动应维护 test basis、testware、测试结果和追踪关系。
  • DORA change approval：高效变更审批应前移到开发过程中的 review，并用自动化检测补充。
  • Scrum Definition of Done：Done 表示增量满足质量标准并处于 usable 状态。
  • Google Engineering Practices：Review 关注设计、功能、复杂度、测试和文档。
  • Temporal Activity timeouts、Celery worker time limits、GitHub Actions limits：长程任务需要总时限、heartbeat/进展观测、资源边界和受控中断。

10. 风险与边界

  • 当前强 gate 依赖 artifact 内容声明和 schema，不能替代真实人类业务决策。
  • reviewProcess.mode 能要求独立 agent/process 证据，但还没有统一外部 process registry。
  • 2 小时以上长超时必须保留 no-progress timeout、人工取消入口和 stdout/stderr/outputDir 进展证据，否则会放大资源占用。
  • QA signoff 现在绑定 targetRef、validatedRef 和最新 qa.validation.ref；后续要绑定 delivery branch SHA 或 PR head SHA，并处理 PR 更新后的重新验证策略。
  • PMO 已有逐节点 pmo.node-checkpoint 审计和末端 checkpoint gate；后续要把节点观测产品化到 CLI/Web，并补风险聚合和人工升级入口。
  • Tekon 仍不自动 merge、不自动上线、不绕过 human gate。
