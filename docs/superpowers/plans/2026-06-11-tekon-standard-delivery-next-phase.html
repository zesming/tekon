Tekon Standard Delivery 下一阶段迭代方案

日期：2026-06-11｜状态：已根据 P1-0 自举失败样本和当前代码约束重新调整

目标：先让 Tekon 以 Codex provider 支持可运行的标准交付种子流程，再逐步补齐独立评审、角色边界强校验、AC evidence、QA final signoff、PMO checkpoint 和长程任务可观测能力。

当前事实

  • P0 已完成真实 Codex 闭环：run_d2350140-b1b7-4fca-b01b-e28daac61e31 创建真实 PR #2，CI 通过，未自动 merge。
  • P1-0 seed run run_04b37267-2686-42c6-a0a4-9b37410f65f7 在 RD 节点 300 秒超时后中断，PM 已产出 demand-card 和 PRD。
  • 当前 artifact 和 gate 都是闭集 schema，首版模板不能直接引用未实现的 independent-review、role-scope、qa-signoff 或 ac-evidence。

调整原则

  • P1-A 首版模板必须 parser-compatible，只使用当前已注册 artifact 和 gate。
  • 角色边界先进入 roles/*/system.md，强制防越权放到后续 deterministic gate。
  • 自举任务必须缩小粒度，避免一个 RD node 同时承担模板、角色、schema、gate 和 PR package。
  • 长程任务采用“总超时 + 进展观测 + 无进展超时 + 可恢复证据”，不只靠等待更久。

阶段总览

阶段 目标 退出标准
P1-0R 归档 seed run 失败证据 失败原因、产物、gate、风险和下一步明确
P1-A0 最小兼容 standard-delivery 模板 模板可加载，preflight 可执行
P1-A1 角色边界模板化 PM/RD/QA/reviewer/PMO 有评审范围、不越权、独立评审、升级条件
P1-A2 长程任务最小支持 真实 provider 默认 1 小时超时，snapshot 保留 timeout
P1-B 独立评审和角色范围强校验 自产自测、越权 review 可被 gate 拦截
P1-C QA signoff 与 AC evidence readiness 不再泛化为 evidence unknown

长程任务策略

层级 能力 判断
L1 默认真实 provider 超时提升到 1 小时，并写入 provider snapshot 本轮落地，降低长程 seed 误杀概率
L2 观测 stdout/stderr、diff、manifest mtime、artifact 文件数和命令活动 后续必须做，否则长超时只是更久等待
L3 node heartbeat、无进展超时、可中断恢复、后台 job registry P1-D/P2 做，不塞进 P1-A0

主要风险

长超时会降低误杀，但没有进展观测会让卡死任务占用更久。P1-A 的角色边界仍是 prompt 层约束，不是确定性强约束。当前 engine 仍按 phase/node 顺序执行，不应把模板理解为复杂 DAG 并行能力。
